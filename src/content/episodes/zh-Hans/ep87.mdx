## 开场：AI 咔嗒时代的不安感    *00:00*

<span class="paragraph-timestamp" data-ts="00:00">00:00</span> **卢正锡** 今天录制的日期是 2026 年 2 月 21 日，周六早晨。上周我们做了申正奎代表的那期，感觉发展速度真的还在变快。四面八方都在涌现“咔嗒咔嗒咔嗒”一天做完、一周做完的高质量软件，我们也有点应接不暇。与此同时也出现了不少声音，说太烦躁了、很抑郁，这个该怎么解决。

<span class="paragraph-timestamp" data-ts="00:38">00:38</span> **崔升准** 我平时不太会直接用“烦躁”或“抑郁”这种表达，但内心确实有那种感受。大概是 OpenClaw 前后吧。随着 Agent swarm、Agent team 出来，我有种自己跟不上那种“跑很多任务”的趋势的感觉，所以 1、2 月我比起快跑，更偏向慢慢思考。但慢下来后背景又在飞速掠过，于是会形成一种 FOMO 的恶性循环。今天也想看看最近发生了什么，虽然我也还没有标准答案，但想聊聊我自己在怎么消化这件事。

<span class="paragraph-timestamp" data-ts="01:15">01:15</span> **崔升准** 我 2 月初在 Uneven Future 演讲时，最后一页是“即使在未知中，也要朝正确方向前进，至少该知道什么？”也就是 MVK，Minimum Viable Knowledge，我们频道也提过几次。我当时就用这个做了收尾，这个问题到现在还一直在我脑子里。我一直在想该怎么逐步知道这些，今天这期应该也会是相关的脉络。

## METR 基准饱和：Claude Opus 4.6 的14小时    *01:47*


<ResourceLink url="https://x.com/metr_evals/status/2024923422867030027" title="METR：Claude Opus 4.6 Time Horizon" domain="x.com" description="50%-time-horizon 约14.5小时——当前 task suite 几乎处于饱和状态" />

<span class="paragraph-timestamp" data-ts="01:50">01:50</span> **崔升准** 先说这个，今天凌晨刚出的消息。METR 公布了 Claude Opus 4.6 的测量结果，结果直接到了 14 小时。我们不是一直在录节目时说，明年大概 5 月、7 月左右可能就会到人类全天 8 小时吗？GPT-5.2 已经接近 5 小时了，4.6 一出来，因为现在是对数图，看上去差异不算大、像是在预期内，但在线性尺度上其实差很多。

<span class="paragraph-timestamp" data-ts="02:25">02:25</span> **卢正锡** 这是直接跳起来了啊。

<span class="paragraph-timestamp" data-ts="02:30">02:30</span> **崔升准** METR 也说了，好像他们的 task 已经 saturate 了，正在准备别的东西。感觉这个图正在走向失去意义的时刻。

<span class="paragraph-timestamp" data-ts="02:41">02:41</span> **卢正锡** 其实这个图的目的本来就是看 AI 在没有人类介入的情况下能独立工作多久。我们一天工作 8 小时，但那 8 小时也不是全程专注，真正在干活可能就 3 到 4 小时。可 14 小时的话它可是纯工作。

<span class="paragraph-timestamp" data-ts="02:59">02:59</span> **崔升准** 对。它有 50% 版本也有 80% 版本，不是单纯比“干得久”，而是对照人类工作时长去测。总之可以理解为：人类 16 小时工作量里，AI 以 50% 成功率在执行。今天凌晨就是以“saturate 了”这个新闻开局的。而且现在看起来“每次加 0.1”成了趋势。原本说会出 Sonnet 5，结果先出了 4.6。上周我还没来得及好好用。因为一直有用 Claude Opus 4.6 的惯性，Opus 还在持续跑，4.6 应该也会表现不错，而且可能更便宜。

## Gemini 3.1 Pro 与不断涌现的新模型    *03:40*


<ResourceLink url="https://x.com/JeffDean/status/2024525132266688757" title="Jeff Dean：Gemini 3.1 Pro 发布" domain="x.com" description="在 ARC-AGI-2 达到 77.1%——相较前作推理性能提升超过2倍" />

<span class="paragraph-timestamp" data-ts="03:46">03:46</span> **崔升准** 然后 Gemini 3.1 Pro 也发布了。基准测试的天花板看起来还是很高，感觉“还能更好吗”，但它又一直在变好，所以以后估计还会更好。Jeff Dean 发的就是现在这个 Gemini 3.1 Pro。可以在 Google AI Studio 进去试，我昨晚跑的一个任务转了 10 小时都没产出。我本来还很期待会出来什么，结果录制前说有错误。性能测试还没完整做起来。总之模型还在不断发布。该怎么评价呢？现在甚至有点麻木了。

<span class="paragraph-timestamp" data-ts="04:32">04:32</span> **卢正锡** 模型其实就是 general 性的终极形态。它具备“什么都能做”的能力。前两天做 Claude Code 的 Boris Cherny 在 Y Combinator 的 YouTube 上说过一句，“最 general 的，反而是最 specific 的”。这股强大的 general intelligence 现在也在横扫其他领域。最近受影响最明显的好像就是科学，你可能也会说到这个。感觉几乎没有它做不了的事。光看着都挺累的。


<ResourceLink url="https://www.modular.com/blog/the-claude-c-compiler-what-it-reveals-about-the-future-of-software" title="The Claude C Compiler: What It Reveals About the Future of Software" domain="modular.com" description="Chris Lattner 深度评测——实现很教科书，但在发明新抽象上有局限" />

<span class="paragraph-timestamp" data-ts="05:04">05:04</span> **崔升准** 当然，日常用起来还是会有卡顿和不顺的地方，但方向上确实是“以前不行的现在变行了”。2 月 18 日，也就是几天前。Chris Lattner 就是做 LLVM、做 Swift、在 Apple 待过、后来去 Tesla、现在创办 Modular 的那位。他本来就是做编译器的，对编译器当然很懂。Claude Code 的 C 编译器出来那天，他发帖说这是他迄今看过编译器相关文档化里最顶级的一批，想再深入看看。后来他把深入看的内容详细写成博客，我也翻译了一下。前半部分他说这个实现有教科书级价值，有不少称赞，当然也有遗憾。整体方向是对的，不是 slop，值得看。尤其因为有 commit 历史，去看它很有意义，类似这样的观点。

## Chris Lattner 对 Claude C 编译器的深度评测    *05:11*

<span class="paragraph-timestamp" data-ts="06:21">06:21</span> **崔升准** 他也指出了哪里做错了。Claude C 编译器最明显的问题是失误，最大的问题是把测试需要的东西硬编码了。这是一个信号：它很难超越测试套件做到泛化。这与其说令人惊讶，不如说是有价值的信息。当前 AI 系统在“组合已知方法并按可测成功指标优化”方面已经很强，但在生产级系统所需的开放式泛化上仍然吃力。

这个观察又延伸到更深的问题，也就是它对 AI coding 本身说明了什么，这部分他讲得很有意思。我很重视这段。看限制的话，“实现已知抽象”和“发明新抽象”是两回事。它确实能消除重复杂活，让你从更接近最新 best practice 的起点出发，起到里程碑作用，但它还不能发明新抽象，这个实现里也看不到新的东西。不过它确实是教科书式实现。

结尾他讲了这样的话：这不是在谈软件的终结，也不是在谈工程师的终结。那有点太 hype 了，反而更像是把门开得更大。实现越容易，真正创新进入的空间反而越大。实现门槛降低不意味着工程师的重要性下降，相反，vision、judgment、taste 的重要性更高了。越容易做出来，越难的是判断“什么值得做”。AI 在加速执行，但意义、方向、责任本质上仍然属于人类。这也很耐人寻味。写代码从来不是目的。目的始终是做有意义的软件。未来属于那些拥抱新工具、质疑既有假设、并设计出让人们可以协作创造系统的团队。

## Andrej Karpathy 的 9 之行进：90% 与 99.99% 之间    *08:35*

<span class="paragraph-timestamp" data-ts="08:23">08:23</span> **卢正锡** 上周申正奎代表来时也说了一样的话。最终软件逻辑才是目的物，代码只是实现逻辑的中间工具，而这个中间工具正在消失。Andrej Karpathy 以前在一次演讲里说过“9 的行进”，用的是“March of Nines”这个表达。把东西做到最初 90% 的努力，把 90% 提到 99% 的努力，再从 99% 到 99.9%、99.99% 的努力，每多一个 9，付出的量级都一样。Claude C 编译器也是，做到 90% 基本能直接给你，夸张一点说 99% 也能给你。但站在 99.99% 位置上的大师眼里，缺的那 0.99% 会是极其巨大的 gap，而这种 gap 更容易体现在你刚才最后总结的那些非常人性的价值上，比如品味、意志。也就是说，我们未来还有很多事要做，大概是这样收束的。


<ResourceLink url="https://wiki.g15e.com/pages/Tasteful%20software" title="有品味的软件 — 姜奎荣" domain="wiki.g15e.com" description="Corca CTO 姜奎荣正在写的文章" />

<span class="paragraph-timestamp" data-ts="09:30">09:30</span> **崔升准** 它有重要意义，但还没到创新那一步，这是 Lattner 的意思。即便如此，这个方向到底意味着什么、当它成为 new normal 时我们能做什么，仍然值得认真思考，我想再次强调这一点。你刚提到“品味”。我们一直反复讲这个词。最近也提过几次，姜奎荣现在是 Corca 的 CTO，好像正在写一篇叫《有品味的软件》的文章。这篇也挺有意思，值得看看。

## 把一切都当作搜索问题：科学与新药研发    *10:12*


<ResourceLink url="https://deepmind.google/blog/accelerating-mathematical-and-scientific-discovery-with-gemini-deep-think/" title="Gemini Deep Think: Redefining Scientific Research" domain="deepmind.google" description="加速数学·物理·计算机科学发现的科学研究伙伴" />

<span class="paragraph-timestamp" data-ts="10:17">10:17</span> **崔升准** 在科学和数学里，不是重复已知问题，而是需要创新。现在那边也在发生很多事。


<ResourceLink url="https://www.isomorphiclabs.com/articles/the-isomorphic-labs-drug-design-engine-unlocks-a-new-frontier" title="Isomorphic Labs Drug Design Engine" domain="isomorphiclabs.com" description="超越 AlphaFold——AI 驱动药物设计的新前沿" />

<span class="paragraph-timestamp" data-ts="10:27">10:27</span> **卢正锡** 这类东西一直不断出现，但我们也看不过来。粗暴总结一下，新药研发本质也是先找出致病蛋白，再看这个蛋白表达某种东西所依赖的机制，然后做出会干预或促进它的物质，也就是常说的 antibody，很多药的机理就是通过化学方式干预这个过程。归根结底，这是识别蛋白结构，再去找与之匹配的另一种结构的游戏。现在它在证明，这些都可以转成 computation，作为搜索问题来解。

## OpenAI 理论物理案例与半人马时代    *11:07*

<span class="paragraph-timestamp" data-ts="11:11">11:11</span> **崔升准** 除了这些，我翻译时认真细读的是 OpenAI 的理论物理论文案例。我去年底就比较强调 OpenAI 与科学这条线，也介绍过 Kevin Weil 和那位做黑洞研究科学家的例子。那位科学家把其他研究者，包括他的导师在内，一些站在理论物理前沿的人带到 OpenAI，一起讨论后，最后出了篇论文。那位加入 OpenAI 的黑洞研究者的导师，一年前还觉得 AI 能帮多少并不确定，哪怕在 2024 或 2025 年初也还是这样。但一年后，在和 GPT-5.2 Pro 多轮往返之后，他们把最终问题提交给 OpenAI 内部模型。那个模型不但解出了量子场论里此前没解开的难题，还给出了证明。只用了 12 小时。模型完成了该领域世界上最聪明的两个人都没完成的事。当我和他们在一起时，他们非常兴奋。大概是这样的叙述。

<span class="paragraph-timestamp" data-ts="12:29">12:29</span> **卢正锡** 是啊。

<span class="paragraph-timestamp" data-ts="12:30">12:30</span> **崔升准** 编码已经开始进入“超越半人马”的阶段，而科学像是刚进入“半人马时代”。Gary Kasparov 输给 Deep Blue 之后，棋界出现了半人马概念。半人马是人马合体嘛。也就是 AI 和人组队会比 AI 单独或人单独更强，这种半人马式下棋曾经流行过。科学终于进入了半人马时代，进入了和 AI 一起做的时代；而编码则走向更大幅度委托给 AI，科学和数学像是在后面跟上。

<span class="paragraph-timestamp" data-ts="13:15">13:15</span> **卢正锡** 感觉所有问题真的都在变成 search problem。投入更多电力和 computation 后，那些没去过的地方现在都由 inference token 去跑遍了。以前要人一个一个思考着做的事，现在似乎都在被替换成 computation。

<span class="paragraph-timestamp" data-ts="13:33">13:33</span> **崔升准** 关于他们这段经历，推特上也有很多争论。不管怎样，这篇博客周边和当事人的说法是：他们认为 2026 年 AI 对物理学将要做的事，会类似于 2025 年 AI 对编码做的事。我把这段翻译发到社媒后，韩国几位科学家也留下了感想，这点很有冲击力。能看到他们在说“原来这种事正在发生，我也得准备，研究生院会怎样”等等顾虑。

## 内插还是外插：模型能力的本质    *14:14*

<span class="paragraph-timestamp" data-ts="14:14">14:14</span> **卢正锡** 这是我们早期就聊过的话题。早期谈模型进展时，我们常说模型只是简单的内插引擎，只能做 interpolation，无法走向新东西，那是在模型还比较笨的阶段常讲的话。可看现在的结果，几乎都相反了。看起来全都是 extrapolation，像在外插。我的看法是，如果模型所投影的整体真理是 100，那这个“100”在模型里是以非常稀疏 sparse 的方式实现的。我还是认为本质是 interpolation。它是在这个广阔空间里，在稀疏区域之间做 interpolation，而在人类视角看起来像 extrapolation。

<span class="paragraph-timestamp" data-ts="15:17">15:17</span> **崔升准** 在极其广阔的区域里做插值，确实可能看起来像新东西，而实际上也会因此产出新东西。

<span class="paragraph-timestamp" data-ts="15:23">15:23</span> **卢正锡** 连我们称为人类独有创造的东西，本质上也可能只是有人找到了早就存在但未被发现的空间，如果把整体普适性当作 100 来看。只要模型的 computation 继续增加、generality 继续增加，那么那些夹在中间的东西也都会被覆盖，虽然可能是时间问题，但都能做到。我们可能得以这个假设为前提。也就是我们在说，模型有可能把这些都做完。

<span class="paragraph-timestamp" data-ts="15:53">15:53</span> **崔升准** 刚才评价是“没什么创新”，但如果把那当作 new normal，我们该做什么？毕竟它做出的事还是非常惊人。我在这个物理学案例里一直像鹰眼一样盯着一句话：“经过数次往返之后”。文中说是内部模型，其他文章还提到 OpenAI 尚未公开的 scaffolding 或 harness。意思是 GPT-5.2 Pro 做不到的，在那个特殊 harness 里连续跑了 12 小时后做到了。我是这么理解的。这个很重要，但“数次往返”也同样重要。为什么他们能做到？当然因为他们是顶级专家、顶级物理学家，但最后终究会还原成某种 prompt。那在还原成 prompt 之前是什么？如果我输入同样 prompt，虽然我可能看不懂，但模型和 harness 应该会给出同样回应吧。这一直是我好奇的点。他们到底用了什么词汇、什么句子。

<span class="paragraph-timestamp" data-ts="17:15">17:15</span> **卢正锡** 这就和你 1 月初说的“好 prompt、会写 prompt”有关。其实每抬一步，模型内部的 space 都在变。也就是 representation 的空间在不断变化，所以才会那样一步步找到路。能够持续迁移那个 space 的能力，也许就是我们隐约感受到的人类价值，也就是品味和方向设定能力。

<span class="paragraph-timestamp" data-ts="17:52">17:52</span> **崔升准** 是的。上周和申正奎聊时，他也透露了一个片段：从去年年中那种写很长 spec 的风格，现在又回到来回对打的模式。我们上期举的是很简单的例子，但申正奎肯定用了融入他自己词汇和知识的 token。

## Agentic 编码的现实：Ralph Loop 的局限    *18:19*

<span class="paragraph-timestamp" data-ts="18:19">18:19</span> **卢正锡** 那当然。最近我也和很多在高强度做 agentic 编码的人交流。以前像 oh-my-opencode 那样，是无条件按目标一路冲到底，不断 hook 让它永远跑下去。但现在好像形成了一种认知：通过这种方式，还是出不来超出我们已知边界的东西，就像刚才 C 编译器案例说的那样。要转向“拓宽地平线”的工作，human-in-the-loop 是必需的。你得理解 context，不断给出 steering 的 prompt，这和 ralph loop 是完全相违背的。

我自己最近做事也有这种感受：特别简单、特别明确、又必须快出答案的任务，就直接跑 ralph loop 粗暴处理。但凡要构建某种逻辑，或者要做特别贴合我们公司业务的东西，就会一小步一小步看。最多也就是开 3 到 4 个子 agent 取回答案，让它们互相比，再看它们比较的过程。这样做完后会发现，input token 量非常大，而 output 很小。可这么做会更容易产出“还不错”的结果，就像刚才说的会有“9 一个个加上去”的感觉。现在 agentic 编码趋势也在变，看那些先行者、先把坑都踩过的人就知道。直到最近之前还是 ralph loop，但现在已经有人说 ralph loop 不行了。感觉可以再请这些前沿的人来聊一次。其实申正奎上周也说了这个。

<span class="paragraph-timestamp" data-ts="20:16">20:16</span> **崔升准** 这也在形成一种 gradient。因为还在持续摸索，如果稍微外插一下，就能预判下一波趋势会是什么。毕竟趋势正在变化。我关注的是“来回对打模式”：一种是高密度词汇和 domain-specific 内容很强的对打模式；另一种是在对打中把东西逐步定下来，形成好问题和指令后，再交给长时 loop 去执行。我感觉这两种纹理会共存并互相增益。

## 商业视角：99 vs 99.99 的差异    *46:23*

<span class="paragraph-timestamp" data-ts="47:04">47:04</span> **卢正锡** 连 layer 都变了。模型现在像一个库一样被封装下沉，我们只要 call 就能拿答案，像一个对象。但要在这上面做商业，不看模型内部就看不清这个产业会怎么走。这是另一点。以前说会 Python 就行，下面 C 或汇编不懂也可以。现在不一样，从硬件 computation layer 到模型，再到商业，你得在不同层级来回切换，才会有答案感。

上周申正奎就说了，他重新开始做“打磨模型”的工作。也说未来可能是：所有软件的单元工作都收进一个模型，再在模型上叠一点 harness。现在我们是挂在 frontier 模型上做 API call，但我理解申正奎的共识是：即便如此，在所有微小细节上，模型世界还是会回来。

到了那一步，这些小模型会成为新的 web 服务。那是把 B2B SaaS 拿来塞一本到手册就 fine-tuning 结束，还是会由那些能把最前沿再推进一层的人取得优势？如果全都在 99、99.9、99.99 之间竞争，客户一定会从 99 流向 99.99。这个变化不是速度感和价值感消失，而是当你回到那个空间后，里面的相对偏差会变成新的 norm 并重新展开竞争。这发生在我们这一代，某种意义上是不幸；反过来说，不管是创业公司，还是迫切想改变的个人，这全是机会。对别人是危机，对我们就是机会。要这样去转换思维。
