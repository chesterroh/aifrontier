## Claude 的「Skills」功能及其工作方式    *08:58*

<span class="paragraph-timestamp" data-ts="08:58">08:58</span> **卢正锡** 另外你在 Claude Code SDK 里真的能体验到，什么都能挂上去然后就能跑。不只是 coding，哪怕只接公司业务或一个数据库，再接 Claude Code SDK，也确实能跑得很好。

最近生物技术或研究相关的 skill package 也像在喷涌出来。对，这个 Harness 现在其实已经成了新的 source code。

<span class="paragraph-timestamp" data-ts="09:27">09:27</span> **崔升准** 对，Claude Code 本身……这不是官方账号，是个分享 changelog 的账号。刚出现。我还以为是 2.1，结果 12 小时前已经上了 2.1.2。2.1 有一次大版本跳升。但几乎每天都在这样持续 bump 版本，这不是单靠人能做到的。Claude Code 又是 Claude 官方做的，由 Claude Code 自己在做，所以功能添加速度非常快，像一种紧急迭代状态。

不过我在年末年初休息时，可能是想回到基本面，就放慢节奏想了一些东西。总之我觉得 Claude 的 skills 很厉害。所以年末我很认真把那些 skills 的 GitHub 文档读了一遍，读得很细。

我以前就会翻译着读那些 skills，这就是那个 repo。比如进到 skills 里，我常用的 algorithmic art skill，翻译后读起来非常有趣，会先讲理念思考，再讲代码前该做什么。翻译着读很有意思。他们说这可能和实际内置 skills 有点不同。就是 GitHub 发布版可能有差异，但这里有很多值得看的内容，我读得很开心。

不过这些 skills 里面有一个叫 Skill Creator 的 skill。我想稍微讲讲它。

<span class="paragraph-timestamp" data-ts="11:16">11:16</span> **卢正锡** Skill 就是把某种能力的 markdown 描述、描述性 prompt 片段、相关代码、可作为示例的内容，按这个文件夹结构打包在一起，我们把它叫 skill。对。

<span class="paragraph-timestamp" data-ts="11:33">11:33</span> **崔升准** 对。OpenAI 去年末也开始用了。现在 skills 像个 marketplace，我觉得已经有几千个在共享了。关键是它真的很容易做，不只在 Claude Code 上容易，就算在 web 界面里聊着聊着也能做出来。我稍后会讲这个想法背景。你在一个上下文里持续对话，会形成一种对话重力，对吧？我现在用的术语、模型回应时用的术语，在这种状态下如果我想把分布拉宽一点，探索另一个空间，就需要引入不同术语，但有时很难想到。

这种时候可以用的方法是：生成四个随机字母为一组，生成 100 组，然后尝试把它们当缩写来读。不一定要完全匹配，核心是利用联想能力，也就是模型的联想能力。有个概念叫 pareidolia，就是人看云会看出动物或脸。人类有联想能力，而现在模型也有很强联想能力。

但要激活这个联想能力，需要输入合适 token。所以我的想法是让它读这些缩写，它就会生成这段 Python 代码。它会实际做出约 100 个随机四字母组合，再挑大概 10 个按当前对话上下文读成缩写。比如把 “Self-understanding is temporary” 读成 “Your Understanding of Self is Temporary” 这种，会出现一些短暂、易逝之类的有趣文本。即使我做不到，模型可能做得到。那它能不能成为一个有意思的分支？因为这样就有 token 了，我们可以围绕它继续对话。所以我在做一些略发散、横截面的探索时会这么试。

然后在这种来回很久之后，我把 Skill Creator 开着了。这个 skill 我就开了 Skill Creator。开太多会分心，所以我只开要用的，通常就常开着。开着聊的时候，它会尝试创建一个 skill。在对话过程中，像刚才那样生成四字母并读缩写，模型先提议做成 skill。我说“要不要做？那就做”，它就建了文件夹结构并产出了这些。每个 MD 文件创建后你点一下，就会一次性复制到我的 skills。流程做得很顺滑。

Claude，所以我……像前面图里介绍过，我一直在说“生成 pipeline 的 pipeline，会产出有用副产物”。对话过程中会产生代码、可复用文本、图片这类副产物。带着这个心智模型去推进事情，总能得到不错副产物。

但我感觉 Claude 已经把这个彻底产品化了。这里还有一轮修订流程，全部做完后我复制到我的 skills，“Domain Priming”，这里我可以输入具体领域术语，也可以不输入。这样它会朝那个方向生成文本，而“尝试去读它”这件事本身就成了一个 skill。你看这里有代码，也有使用示例，所以现在 Claude 可以在需要时调用它。可以用了。而它在我的 skills 里，我可以开关。

<span class="paragraph-timestamp" data-ts="16:01">16:01</span> **卢正锡** 对，做一个 skill 其实很容易。我们可以在自己的 Claude 里用，同样结构在 Claude Code 里也能一直建 skill 并扩展。就是这样。

<span class="paragraph-timestamp" data-ts="16:13">16:13</span> **崔升准** 对，所以你只要把文件夹原样放进去就行。

## Skills 为什么效果好：精确 token 的重要性    *16:17*

<span class="paragraph-timestamp" data-ts="16:17">16:17</span> **崔升准** 但 skills 为什么会这么好用？

<span class="paragraph-timestamp" data-ts="16:22">16:22</span> **卢正锡** 我也不太确定。

<span class="paragraph-timestamp" data-ts="16:23">16:23</span> **崔升准** Skills 其实建立在去年很流行的 spec-driven 方法之上，有这个基础。再按某种格式，用 MD 来组织。现在 MD 文件最终也成了一种编程，MD 结构做得非常完整，而且不只是 MD，还有带代码的脚本。所以它不完全是静态，而是……既有精确执行的部分，也有概率执行的指令和代码。把这些都打包得很整齐，便于复用。

所以在当前形势下，MCP 也是个话题，但我感觉 skills 正在变得非常重要，而且做得很工整、很好。但为什么这种 skill 必然会有效？我现在还没有答案，但至少这个问题已经浮现。去年底到今年初我有时间想这个。

我把这个标成 side path，既然是 side path 也许不展开，但 Boris Cherny 还开源了一个 code-simplifier。Anthropic 发布的。你看会发现很简单，就是一个 MD 文件，指导你如何有意义地总结代码，是一套指南。现在这类东西就是插件，正在这样不断堆叠。

但如果把它的注意力转到特定领域，在那里给它触发器，像在说“我现在要做这个了”，你得先铺垫。那它在相关任务上就会好非常多。


<span class="paragraph-timestamp" data-ts="19:39">19:39</span> **崔升准** 会好很多。我做了很多测试，也在群聊里跟别人一起做过。用特定术语和不用术语，输出质量差异挺大。比如只是大概套一个专业 persona 来说话，和真正输入真实专家会用的 token，至少是相近 token，回答质量会变。但从 Transformer 原理看，这其实很自然。

## 不是专家也能使用专业术语的方法    *20:13*

<span class="paragraph-timestamp" data-ts="20:13">20:13</span> **崔升准** 那问题来了，这样的话最终是不是领域专家更占优？你会这么想。是不是懂的人就用得更好？

但 AI 发展方向真的是这样吗？如果要成为人人可用的 leverage，模型应该能把人带起来。然后像 skill 那种 prompt 上下文里，如果你用了合适 prompt，即使我不是这个领域专家，也能把该领域知识拉出来用，甚至把模型过量能力作为 overhang 拉出来用，那就更好了。
