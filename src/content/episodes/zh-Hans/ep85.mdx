## 从 Ralph Loop 到迭代式 PRD 文化    *00:31*
<span class="paragraph-timestamp" data-ts="00:31">00:31</span> **崔升准** 我们说过 2 月一到一定会有事发生，结果真的发生了。看起来像是计划好的一样。我能感到范式在转移。

如果把时间往回拨，去年夏天 Ralph Loop 很火。还有 Oh-My-Opencode，那些都是围绕着选定一个 PRD，然后反复执行直到做出来的那种流程。所以你会烧掉很多 token，不管怎样先产出点东西，PRD 写得越好，完成度就越高，这类模式出现了。

所以借用了《辛普森一家》角色 Ralph Wiggum 那种不断试错但最终做成的感觉。做这个的人 Geoffrey Huntley，先写了一篇关于 deliberate practice 的文章，然后宣布并介绍了 Ralph Loop。

## OpenClaw 哲学：Human In The Loop vs 完全自治    *01:23*
<span class="paragraph-timestamp" data-ts="01:23">01:23</span> **崔升准** 但 OpenClaw 成了很大的话题。看 OpenClaw 创始人 Peter Steinberger 的采访，他的视角稍微不一样。

与其像 Ralph Loop 一样让模型自己不停重复直到成功，Peter 更偏好 Human In The Loop。Peter 有他自己的哲学，核心是：在不做安全沙箱限制的状态下，让模型使用大量工具，通过给它们完整权限来把事做成，并且通过消息工具来完成，再把这些东西投放到市场，引发了巨大震动。

## Pi 与 VibeTunnel：在消息工具里使用 agent    *03:02*
<span class="paragraph-timestamp" data-ts="03:02">03:02</span> **崔升准** 还有 OpenClaw 在 agent 内核层面其实用了一个叫 Pi 的东西。所以我也带着兴趣去看了 Pi，做它的人是 Mario Zechner，这位就是 Mario。做 OpenClaw 的 Peter 好像和 Mario 很早就认识。然后去年 Claude Code 出来之后，欧洲维也纳那边有个 hackathon，Mario 作为团队成员一起做了一个叫 VibeTunnel 的东西。他就是参与了那个。

那是什么呢？是让你在消息工具里使用 Claude Code。也就是说，你可以在任何地方让强大的 agent 处理你本地机器上的事情。但如果不是必须写代码，而是能像自然语言协作一样处理代码，关键点就像是在大幅降低门槛，因为不然很多人根本用不了。这个方向看起来是重点。所以这种哲学也部分进入了 OpenClaw，把各部分连起来了，特别是在 Discord 这类场景里，它能让你展开并做很多事。

Pi 的哲学是维持非常精简的功能集，甚至连 MCP 都不加。它会加 skills，但整体方向是让 agent 自己去构建那套软件。所以是最小化、正交化功能，通过组合积木来构成一切的哲学。关于这整套我还没积累足够经验，目前只试到安装 Pi 这个程度。

<span class="paragraph-timestamp" data-ts="04:32">04:32</span> **卢正锡** 所以 Pi 在 OpenClaw 里，可以把它看成核心 agent harness，对吧？

<span class="paragraph-timestamp" data-ts="04:38">04:38</span> **崔升准** 对。它是中心核心，而 OpenClaw 就是在上面打了个 Stim Pack。

所以像我们前面说的，一种做法是通过 deliberate practice 无限让它做下去，另一种哲学是先拿最小功能再展开；更激进一点的版本是连 sandboxing 都不做，但依然强调 Human In The Loop，这就是 OpenClaw 的路子。然后又出现了像 Moltbook 这种游乐场空间。

不过有个有趣的共性是，这四个人全是有 20 到 30 年从业经验的资深“老”开发者。都在 40 到 50 岁，有过 exit 经验，各自有工程哲学和品味，把这些提炼后做成了自己的工作 harness。

## Agent Teams：共享任务列表与可观测性    *12:30*
<span class="paragraph-timestamp" data-ts="12:30">12:30</span> **崔升准** 那篇用并行 Claude 团队构建 C 编译器的帖子非常震撼。就是用 agent swarm 并行来做，文中还出现了“running the Ralph Loop”这个表述。做到可行为止，直到 C 编译器做出来，他们试验了自动驾驶式推进。那个视频，Claude team、Claude Code team 的视频，是 Lydia Hallie 上传的。没有声音，只看流程。

总之，Claude 公告里重要的是那些会留下工作痕迹的文件、在 MD 文件里管理死锁、以及何时释放和获取锁的重要含义。他们跑了 16 个 agent 持续很久，最后给出了一份关于构建 Rust-based C 编译器的总结。这个也很惊艳。至于 Opus，也就是 Claude 团队，Anthropic 去年讲的是主 agent 编排 sub-agent。

它会不断产出 result、result、result，然后汇报，主 agent 也就是 main agent 的可视管理负担很重。但现在方向变了。Agent Teams 当然也有 lead/main agent，也有任务分配，但它有共享任务列表，不是各自分派，而是更强调委托和队友间互相沟通，围绕共享任务列表形成上下文。通过反复在这个空间里做事和沟通，lead/main agent 的负担会显著降低。

这篇里说它越来越像组织文化，我理解是这个意思。Cursor 团队做的是一篇叫“Toward an Autonomous-Driving Codebase”的文章，展示了数千个 agent 的执行，我读得很过瘾，和前者类似但又是另一条路径。

我这里强调的是，这一切都说明“抽取、陈述并理解意图”有多重要。意图非常关键。所以当你让 agent 在这个规模上工作，这种重要性会更高。Steerability 和 observability 仍会是很有意思的研究方向。所以在很多团队里，不是很小的那种，而是超大规模团队里，如何设定方向、观察团队在做什么，以及为了做到这一点，明确抽取和定义意图并让 agent 理解它，这一点让我印象很深。

<span class="paragraph-timestamp" data-ts="15:37">15:37</span> **崔升准** 对我来说有趣的是，在人类社会里，如果我们学习了 agent 社会，就能验证在 agent 社会里到底什么真正有效。然后那会反过来反馈到人类社会。

<span class="paragraph-timestamp" data-ts="15:53">15:53</span> **卢正锡** 结构是一样的。我会说它是同构的。

<span class="paragraph-timestamp" data-ts="15:56">15:56</span> **崔升准** 这篇最后的结论是：taste。我们去年也聊了很多 taste，对吧？taste、判断和方向来自人类，但在这个研究里，AI 是快速迭代和探索的强力放大器。起点来自人，但要真正完成巨大成果，含义是 AI 把这种力量放大了。

所以这两者一起存在，而且也已经和 AI 一起实践过了。它们有哪些共性和差异，如果你感兴趣，我觉得值得一看。
