## 前沿模型的爆发式增长与中国崛起    *01:14*

<span class="paragraph-timestamp" data-ts="01:14">01:14</span> **金成贤** 与学术侧分开看，或者说有重叠的一点是，最有意思的是大量开放前沿模型的出现。我觉得这是最大的点。DeepSeek、MiniMax、Z.ai、小米、腾讯、Moonshot、蚂蚁、阿里巴巴、美团等等，很多公司都发布了模型。以前这些模型大概是 Llama 2 的规模，70B 左右或最多 70B，做个模型再发布是常见形态，但 2025 年发布的模型几乎都是 frontier 或 near-frontier 级别，而且这些公司大多都在瞄准 frontier。我觉得这是 2025 年最大的变化之一。

比如在 2024 年，更像是“在现有资源内能做什么就做什么”。与其说是做 frontier 模型，不如说对所谓小模型、效率模型的关注更多。也可以说那就是当时的上限。看起来确实是那样，但到了 2025 年，这类模型明显变少了。

几乎所有发布模型的大公司都在瞄准 frontier 级别，追求更大、更强的模型。我认为这代表了非常显著的思维与趋势变化。

<span class="paragraph-timestamp" data-ts="02:44">02:44</span> **卢正锡** 你刚列的这些模型是不是 100% 都是中国的？

<span class="paragraph-timestamp" data-ts="02:46">02:46</span> **金成贤** 对，100% 都是中国的。中国以外几乎没有可以称为 frontier 且给出惊艳结果的模型。像 Llama 4 这种有，但 Llama 4 没留下太深印象。到年底 Mistral 说要发布模型，但看起来他们的 frontier 级模型也不是 open model。所以目前是中国在领跑。

而且中国在相对受限的算力条件下，依然在瞄准 frontier，想往下一层级走。我觉得这本身就是很大的范式变化。大家对中等偏小模型的兴趣已经没那么高了。几乎都在追求高性能、大模型。我认为这是 2025 年最大的变化之一。很多不同公司都在造模型。

但现在变了，是因为已经被证明：即使算力不大，也能冲击 frontier 级别。我觉得这是变化发生的原因。这里面最大的玩家还是 DeepSeek。DeepSeek 证明了在有限算力下，大约 800 到 2,000 单位算力，也能瞄准 frontier。

一旦这点被证明，大家就开始转向“冲 frontier”。转折点就是那时出现的。所有人都开始往更大、更强的模型走。就是这样开始的。

## MoE：更少计算，更高性能    *04:34*

<span class="paragraph-timestamp" data-ts="04:34">04:34</span> **崔升准** 这张图该怎么读？

<span class="paragraph-timestamp" data-ts="04:36">04:36</span> **金成贤** 其实我很犹豫要不要放这张图。直觉上理解的话，最浅蓝那部分可以看成所谓 dense model，上面这些线是 MoE 模型。可以这样看。图例这里有点错，这些是 MoE 模型。

这张图很重要、冲击力也很大，因为横轴是训练计算量。训练计算量在 10^24 左右时，已经接近 frontier 模型但略低。在这个计算规模下，MoE 模型的性能比 dense 模型好 7 倍以上。也就是说，同样是 10^24 compute，MoE 的表现接近于一个投入了约 7 倍 compute 的 dense 模型。性能可以做到类似水平。

所以如果你给 dense 模型 10^24 compute，再用同样的 compute 做 MoE，那么 MoE 会给出接近 7×10^24 dense 的效果。这很有冲击力，因为训练 compute 越大，这个倍数还在变大。这是很少见的现象。模型里通常能保持线性翻倍就很了不起了，但目前我们看到的是，MoE 下随着训练 compute 增加，这个乘数还在继续扩大，也就是越往后越好。相对 dense 来说，已经到了“不用 MoE 反而奇怪”的程度。

<span class="paragraph-timestamp" data-ts="06:14">06:14</span> **崔升准** 但这里为什么有三条不同的 MoE 模型？为什么分开？

<span class="paragraph-timestamp" data-ts="06:20">06:20</span> **金成贤** 这是所谓 sparsity。即总参数量与单次推理 token 实际激活参数量的比例。

<span class="paragraph-timestamp" data-ts="06:32">06:32</span> **崔升准** 就是说像切成更小块那种感觉？

<span class="paragraph-timestamp" data-ts="06:35">06:35</span> **金成贤** 对，比如这一条每次只用总参数的四分之一。

上面那条则是每次只用总参数的五十分之一。参数用得越少，也就是越 sparse，这个 multiplier，我该怎么说，叫 compute multiplier，会更大，斜率会更陡。

当然这是按 FLOPs 算的，所以考虑真实推理条件、内存带宽等，不一定完全如此。但只看训练 compute 维度，这种模式出现本身就很令人惊讶。

<span class="paragraph-timestamp" data-ts="07:20">07:20</span> **崔升准** 它不完全是 scaling law，但感觉很像。

<span class="paragraph-timestamp" data-ts="07:24">07:24</span> **金成贤** 它是 scaling law 的一种。这个 scaling law 也有两个组成：一个是指数项 exponent，一个是前面的系数 coefficient。当模式体现为 exponent 在变化时，就会出现这种现象。那这种情况下，不用它反而不合理，它会变成“必须采用”的技术进步。MoE 就扮演了这个角色。

到 2024 年前，MoE 模型还比较少，但现在 2025 年出来的模型几乎全是 MoE。除了 edge device 这类场景，基本都是 MoE，像 GPT-OSS 也是非常 sparse 的 MoE。

而真正把 MoE 的 recipe 建起来的，DeepSeek 贡献巨大。某种意义上，DeepSeek 设计的架构变成了上一代 Llama 架构那样的基础架构。比如 Kimi，Moonshot 的 Kimi，就说没必要再去改 DeepSeek 架构了。因为这套已经足够好，直接拿基础结构跑就行。他们做了很多几乎原样采用的工作，我觉得 Mistral 可能也做了类似的事。DeepSeek 开创的 MoE 架构就是这么成熟。通过这个，大家都意识到：即使算力非常有限，只要采用这套架构，也能去冲 GPT-4 以上的目标。这是非常关键的组成部分。

## MoE 如何工作，以及 DeepSeek 的贡献    *09:01*

<span class="paragraph-timestamp" data-ts="09:01">09:01</span> **卢正锡** 直觉上讲，MoE 会让人觉得“嗯，这很好”，但同时又有点理解不了，是这种领域。

我们说 expert，很容易想成这个 expert 负责数学、那个负责科学，但其实并不是这样。

每个 token 都会走不同的 expert routing，其中还有共享的东西、同时激活 7 个之类，全部都是超参数配置。

成贤，MoE 现在有被揭示出来的理论背景吗？比如它为什么有效。

<span class="paragraph-timestamp" data-ts="09:37">09:37</span> **金成贤** 对最近的 MoE 模型，最好把它看成 sparsity。这样理解最合适。用 sparsity 这个概念想非常好：虽然可能有上百个模块，但每次真正用的只是其中一部分。这就是稀疏状态。

通过这种稀疏性，每次真正消耗的计算量被限制住了。因为每次实际激活参数只是总参数的一部分，这部分是固定的。

总参数又非常大，而且每次根据情况、根据 token 会调用不同模块，所以从全局看总参数会产生一种放大效应。这个解释可能有点难。

<span class="paragraph-timestamp" data-ts="10:26">10:26</span> **卢正锡** 对，很难。确实很难解释。

<span class="paragraph-timestamp" data-ts="10:30">10:30</span> **崔升准** 有点模糊，但是否可以理解为模块化、正交化之后可组合？

<span class="paragraph-timestamp" data-ts="10:37">10:37</span> **金成贤** 对，就是这样。把“可组合性”打开的也是 DeepSeek。它被模块化了，每次只用其中一部分。虽然取决于场景，但因为准备了大量模块，从系统整体看，就产生了一个超大模型的效果。可以这么理解。

在 23、24 年很多人都觉得 MoE 是好方向，但我不认为有人预期它会这么好。

即使 GPT-4 是 MoE 这件事已经公开之后，随着经验积累，感受也从“不错”变成了“这是不是好得过头了”。

<span class="paragraph-timestamp" data-ts="11:13">11:13</span> **崔升准** 所以这是 2025 年第一名。

<span class="paragraph-timestamp" data-ts="11:15">11:15</span> **金成贤** MoE 是第一名。现在这些模型都换成 MoE 了，非 MoE 反而很少见。

<span class="paragraph-timestamp" data-ts="11:23">11:23</span> **崔升准** Chester 评测的 Kimi k2 里，MoE 分出来的单元很多，是几千个吗？

<span class="paragraph-timestamp" data-ts="11:30">11:30</span> **卢正锡** 不是，大概几百级别。比 DeepSeek 增加了数量，但整体架构一样。正如成贤刚说的，他们应该也做了“与 sparsity 相关的效率在什么水平最优”的实验，做过一些这种尝试。

## RLVR：开启 Agent 学习新视野    *12:01*

<span class="paragraph-timestamp" data-ts="12:01">12:01</span> **金成贤** 下一个就是 RLVR。它几乎把 o1 的方法完全揭示出来，带来了巨大变化，而这最终也是 DeepSeek 做出来的。

<span class="paragraph-timestamp" data-ts="12:14">12:14</span> **卢正锡** 完全同意。

<span class="paragraph-timestamp" data-ts="12:15">12:15</span> **崔升准** 现在回想起来真的很惊人。

<span class="paragraph-timestamp" data-ts="12:18">12:18</span> **卢正锡** 他们做得太好了。24 年底 DeepSeek-V3 论文出来，定义了 MoE 等内容；到了 2025 年 1 月发布 DeepSeek-R1 模型，RLVR 作为方法论随论文一起出来。

<span class="paragraph-timestamp" data-ts="12:35">12:35</span> **金成贤** 其实 DeepSeek 的影响反而被低估了。它把范式和市场格局都改了很多。RLVR 通过可验证正确答案来给奖励，让 reasoning 学习成为可能。2024 年大家探索的很多东西里，它用几乎最简洁的方法给这个核心路线画上了句号。也因为它给了 o1 方法很强线索，基于这个，o1 类 reasoning 模型才变得可开发。

reasoning 本身大幅提升了模型性能，也打开了所谓 agent post-training 的一条路径。reasoning 不只是“苦想数学题”，而是模型用工具、与外部环境交互，并在交互中作为 agent 完成任务。

这个过程整体都可以放进 RLVR 的奖励框架里训练，模型就会开始以 agent 方式学习。这影响非常大，已经影响了当前市场。现在出来的 frontier 模型，与过去“先做 base pre-training，再做一点 post-training、做点 instruction-following 就能发布”的时代不同了。

现在 agent post-training 和 reasoning 能力几乎成了标配。这带来了巨大变化，不只是可用性，也极大拓展了潜力。agent 模型对现实市场的影响已经无需赘述。今天的 coding agent 和各种 agent，都是这种 agent post-training 才能实现的结果。

<span class="paragraph-timestamp" data-ts="14:25">14:25</span> **崔升准** 那我这样说会不会太夸张？如果 RLHF 是为了做聊天机器人的 post-training，RLVR 就是为了做 agent 的 post-training。

<span class="paragraph-timestamp" data-ts="14:35">14:35</span> **金成贤** 对，通过 RLVR，模型被训练成 agent 成为了可能。可以这么看。

<span class="paragraph-timestamp" data-ts="14:43">14:43</span> **卢正锡** 这个话题上次我们和成贤深聊过，真的很有意思。更好奇的朋友可以去听成贤上一期。

<span class="paragraph-timestamp" data-ts="14:56">14:56</span> **崔升准** 但那时候你还没用“agent post-training”这个说法吧。

<span class="paragraph-timestamp" data-ts="15:01">15:01</span> **金成贤** 对，当时没讲 RLVR 如何连到 agent。比如说，模型要成为 agent，必须给它各种工具。比如调用工具使用编辑器，或者在代码场景里读仓库文件、写入修改，这些都通过工具完成。模型借助工具和外部世界交互。那在这个过程中，如何让模型把这件事做好、如何训练？针对“如何用工具完成目标任务”这个问题，RLVR 给了很简洁的答案：先让模型能用工具，至于中间怎么用先放一边，按最终结果评估。范式转到这里。

比如一个 coding task，就评估最终产物是否令人满意。unit tests 就是典型例子。通过 unit tests 可以验证代码是否正确，验证最终结果。至于达到结果的中间过程，人类可以暂时不去管。

模型自己去用工具，最终设法达到结果。达成目标就给奖励。这样就发生了 agent post-training。当然这只是很简化的说法，还有 cold start 等问题，但核心就是这样。通过只评估最终结果，模型被训练成 agent 成为了可能。过去这些过程都要人工设计。

在你上次那期里你讲到 RL，尤其 RLVR 到底产生什么影响，你当时的一个反思让我印象很深。很多能力其实 baseline 本来就有，但通过 RL 能更容易把这些能力调出来。这是我最大的收获。

## 深入 RL：能力涌现还是能力组合？    *18:22*

<span class="paragraph-timestamp" data-ts="18:22">18:22</span> **金成贤** 这个和模型性能稍微不同，但关于 RL 很有意思的一点是 RL 基础设施的发展。LLM RL 的要求非常复杂。因为你要训练模型，需要训练基础设施；你还要用模型生成并与环境交互，所以需要采样与生成引擎；再把采样结果送到环境中获取反馈，这种基础设施也要有。

而且这些设施彼此联动。训练结果要给采样引擎，采样结果进环境，环境结果再回到训练。单看基础设施层面，这三件事任何一件都很难。仅仅训练这一项，要高效做好就已经极难，这几乎就是 pre-training 基础设施的全部难点。

但现在还要让这些异构系统联动。模型训练要快，采样生成也要快，训练与生成切换要快，评估和奖励要快，而且全流程还必须准确。这个过程稍有误差，就有很多研究表明会妨碍学习。

在 2025 年，“又快又准”始终很难，但这带来了很多新问题。再加上 MoE 成为主流后，如何稳定做 MoE RL、如何稳定训练 MoE+RL，也变成重要议题。这很难，但也许因为中国团队集中攻关，推进非常快。我觉得这是 2025 年非常重要的一根支柱。对 RL 的理解也推进了很多。

理解 RL 的一个轴是方法改进。RL 到底如何发生？RL 发生时 LLM 内部到底变化了什么？为什么 RL 在 LLM 上效果这么好？这类研究很多。

但在“理解 RL”这件事上，我个人很感兴趣的一项研究是：LLM 和 RL 刚火起来时，就有个问题一直存在，RL 到底是在赋予新能力，还是只是把已有能力调出来。也就是能力本来在那，只是被埋住了，RL 把它提上来。早期常见观点是“可能基本就是这样”。

但随着理解加深，出现了“RL 会带来新能力”的观点。这里有 atomic skill（原子能力）和把原子能力组合的能力。原子能力可以理解成四则运算这种；而把这些能力组合起来解更复杂问题，是 composition 能力。原子能力重要，组合能力也同样关键。现在很多人的看法是：原子能力在 pre-training 学到，比如算术；而 RL 学到的是把 pre-training 学到的能力进行组合的能力。

<span class="paragraph-timestamp" data-ts="21:57">21:57</span> **金成贤** 和 MoE 还是不太一样。这里是说算术这种更基础能力。大家认为 RL 不太擅长直接学这种基础能力本身。

但“把基础能力按正确顺序组合，去解决新问题”这种 compositional 能力，是可以通过 RL 学到的。就现在来看，这是 2025 年里理解 RL 最有意思的话题之一。

<span class="paragraph-timestamp" data-ts="22:28">22:28</span> **崔升准** 听起来像 skill 1、2、3 这样逐步推进，这一块似乎有点不同。这个很重要吗？

<span class="paragraph-timestamp" data-ts="22:36">22:36</span> **金成贤** skill 1、2、3 都可以看作独立技能。不仅在这个问题里，在其他问题中也会出现。一般认为这些技能在 pre-training 阶段学到。单项技能在 pre-training 学到，但要真正解题，必须把这些技能组合好。

<span class="paragraph-timestamp" data-ts="22:58">22:58</span> **崔升准** 这个 composition、combination，是不是有点像 chaining，一个接一个串起来？

<span class="paragraph-timestamp" data-ts="23:05">23:05</span> **金成贤** 对，chaining 是 composition 的一种形式。拿这个技能的结果接下一个，再接下一个，就是这种组合方式。

比如简单算术题，只靠组合四则运算也能做很多任务。所以虽然算术单项技能存在，但“如何组合这些技能”也可以视为另一种能力。

而且常有人说，这种 compositional 能力似乎是 RL 赋予的。通过这个理解，我们对 pre-training、mid-training、post-training 的关系理解提升很大。

当然 atomic skill 到底是什么、该怎么定义，仍然微妙。说 atomic skills 在 pre-training 学到，但“原子能力”到底是什么？可能不只四则运算这种，也可能是更大粒度，这方面仍有很多待理解。

尽管如此，我们已经获得了很多关于“模型如何提升、RL 具体做了什么”的理解。如果 RL 带来的是组合能力，那么 pre-training 阶段就应当尽可能培养更多 atomic abilities，把基础能力打磨好。这样模型会更强，也能做更多事。这种理解就出现了。

<span class="paragraph-timestamp" data-ts="24:36">24:36</span> **崔升准** 这是经验观察吗？为什么 RL 会获得新的、我是说 compositional 能力，这一点还不知道吧？

<span class="paragraph-timestamp" data-ts="24:43">24:43</span> **金成贤** 对，还没有理论分析到那个深度。主要是经验上，在一些简化设定里得到了结果。比如模型基础能力不足时，RL 本身效果不好；如果基础能力存在，那么通过 RL 能获得更长链条、更复杂组合的能力。这些都被经验验证了。

## Self-play 与内在动机：与人类对齐的 AI 条件    *53:10*

<span class="paragraph-timestamp" data-ts="53:10">53:10</span> **金成贤** 与 RL 相关总会出现一个词：self-play。AlphaGo 通过 self-play 提升性能这件事给大家印象很深。

所以大家对 self-play 也很期待。这也与数据问题有关。希望模型不用外部喂数据也能自己学。

但我们现在做的数学题或 agent coding，不是围棋那种游戏。它不是零和博弈，所以 self-play 很难实现。

比如可以想象：一个 agent 出题，一个 agent 解题。出题 agent 不断出更难题，解题 agent 不断去解，通过交互让模型变强。

那出题 agent 会持续把题变难，越难奖励越高。

但这里有陷阱。围棋我不确定，但数学题里，制造“成功率 0%”的题太容易了，胡乱出题就行。

于是会说，那别出 0% 的题，出“刚好合适”的题，比如 50% 成功率。

但这也很容易做。比如四则运算，你只要不断拉长算式长度就能调难度。

这说明：如果我们关注“人类真正觉得有趣的问题”，让 self-play 起作用其实很难。关键不是把成功率压低，不是单纯出更难题，而是要出人类视角下有趣、有价值的问题。

这是个非常难的问题，也有很多研究者在做。最近论文里的倾向是：如果不与人类对齐，这件事很难成立。模型不与人类对齐就走不通。很多人都在往这个方向思考。
