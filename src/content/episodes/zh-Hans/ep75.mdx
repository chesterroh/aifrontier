## 后训练时代与模型开发周期    *01:14*

<span class="paragraph-timestamp" data-ts="01:14">01:14</span> **金成贤** 所以新模型发布周期越来越快。现在我们正从 pre-training 时代进入 post-training 时代，各家公司都在改进自己的 post-training recipe，这些改进后的 recipe 会被直接注入模型。相比先改进 pre-training、再做 post-training、再发布新模型的速度，现在改进模型发布得快得多。这个周期正在变得非常快。GPT-5，或者 OpenAI、Anthropic 的模型，这个周期可能还会继续加速。他们也在引入各种东西，比如单条指令可以执行 200-300 次 tool use 等等。

他们说分数好、性能好，除此之外还有个有意思的点：Kimi K2 以及之前的 Kimi 模型，不只是强调 coding 和 math。你会看到他们一直在强调 creative writing 和写作能力，这一点强调得很多。

我自己用的时候，也确实会感觉 Kimi K2 和 Moonshot 模型有一种属于 Moonshot 的写作风格或语气。写作这方面也很有意思，而且他们也在持续强调并延续通用能力。我觉得这些点很有趣。尤其最近常说的“spiky intelligence”又被提得很多，比如只擅长 math 和 coding、或者只擅长特定领域的模型，大家对这些方向的兴趣在上升。但另一方面，前沿公司对通用智能和多任务能力仍然保持很高兴趣，也在持续推进。

## 通过 MoE 与量化实现模型轻量化    *02:55*


<span class="paragraph-timestamp" data-ts="02:55">02:55</span> **金成贤** 再具体一点，他们提到了“inference efficiency”，这个 inference efficiency 指的是服务效率。谈到服务效率时，他们说在 post-training 过程中做了 quantization-aware training。也就是这么说的。quantization-aware training 是一种在把模型做量化缩小时，尽量减少训练阶段性能下降的技术。这里写的是在 MoE 部分，也就是 MoE FFN 部分做了 INT4 量化。和 OpenAI 的 gpt-oss 类似，做了 4-bit 量化。

他们用的是 MXFP4 形式的 4-bit 量化，同样是在 post-training 阶段引入 INT4 量化，这些趋势看起来越来越主流。这可能还需要结合 MoE 研究进一步深入看，但很多讨论都认为 MoE，尤其是 MoE 部分，相比更一般的 dense model 更容易量化，效果也更好。

这很自然，因为模型训练越多，模型权重里信息压得越多，量化就越难。但 MoE 的情况是模型每个模块并不是在全部数据上训练，而是部分训练，所以可以说压缩潜力更高。也因此 gpt-oss 特别量化了 MoE 部分，现在 Kimi 也在量化这个部分。所以这些趋势以后大概率会变成我们越来越常见的基础技术。

<span class="paragraph-timestamp" data-ts="04:38">04:38</span> **卢正锡** 那这样模型体积应该缩了很多。这是 1 trillion parameter 级别模型的话，体积应该不到 1TB。理论上会在 500GB 左右？大概 600GB？

<span class="paragraph-timestamp" data-ts="04:51">04:51</span> **金成贤** 因为大部分权重都在 MoE 部分，应该不会到 1TB 级别，不会那么大。当然量化后加载进内存不会是 1TB，像你说的，大概 500GB 左右会加载。

## 关于强化学习（RL）的新视角    *07:26*

<span class="paragraph-timestamp" data-ts="07:26">07:26</span> **金成贤** 好，那我回到我原本要讲的强化学习。上次讲了 RLVR，讲完后我原以为那样就够了，但实际做完后我强烈感觉应该讲得更深、更好。所以这次我再讲一次强化学习。对我这一代工程师来说，强化学习常常很特别。因为我们从 AlphaGo 开始。也就是 deep learning 的那波，所以很多人第一个项目可能就是复现 AlphaGo。

但相比之下，我不是从强化学习开始的，我只是从监督学习开始。对强化学习我以前会觉得“这么麻烦，为什么要做？”一直到最近都这样，但 LLM 时代到来后，我没法再回避强化学习，最后还是讲到了它。

所以从这个角度看，和那些一开始就做强化学习的人相比，我对强化学习的视角或感受会有点不同。对那些长期热爱强化学习的人来说，我最初是把它当成麻烦事，是出于必要性才重新进入这个领域。作为后来再次对强化学习产生兴趣的人，我的视角可能有些不同。不过我觉得这种新视角可能也有意思。

先简要再介绍一下什么是强化学习：有一个 agent，agent 在 environment 里采取 action。采取 action 后，environment 会因为这个 action 发生变化，我们称之为 state 变化。某些情况下会给一个 reward，也就是在这个 environment 里定义的奖励。比如游戏里得高分就是一种 reward。于是训练这个在环境里行动的 agent 去最大化 reward，这种学习方式就是强化学习。

## 监督学习 vs 强化学习：自动驾驶类比    *09:30*

<span class="paragraph-timestamp" data-ts="09:30">09:30</span> **金成贤** 这部分和监督学习有点不同。不过监督学习当然也不是和强化学习完全割裂。把它们完全分开看有些困难，因为监督学习的一些技术也可以视作强化学习的一部分。

但通常说强化学习和监督学习的差异时，关键点是：不会由人类专家直接教 agent“应该怎么做”。不会这么教。

<span class="paragraph-timestamp" data-ts="09:56">09:56</span> **崔升准** 有点容易混淆的是，这听起来不是更像无监督学习吗？不是监督学习里的 RL，而是无监督学习里的 RL？

<span class="paragraph-timestamp" data-ts="10:04">10:04</span> **卢正锡** 说成监督学习也没错。成贤的意思是，强化学习推进时，对成功的部分做 reward update，这个过程其实和监督学习里按标签学习有点像。

<span class="paragraph-timestamp" data-ts="10:21">10:21</span> **金成贤** 这里用自动驾驶做类比会比较好。如果用监督学习训练自动驾驶，人会先做出驾驶轨迹，记录驾驶过程，然后让模型去模仿这份记录，这更接近监督学习。

如果用强化学习，你给它的是目标，成功到达目标、到达目的地会得到 reward。至于怎么开，让 agent，也就是 AI model 自己找方法。这是监督学习和强化学习比较时最大的区别。它不会教你具体怎么解题，更像只给你目标：“把问题解出来。”而这恰恰带来优势。

如果用人类造的数据训练模型去模仿，最终通常就是人类水平。当然不能说永远如此，但一般目标就是达到人类水平，因为它在模仿人的方法。实际上很可能比人还差，模仿通常就是这样。

但强化学习让模型自己找方法，于是就有超过人类水平的可能性。围棋就是典型例子。因为模型自己找到了下棋方法，所以达到了超越人类的水平。从这个意义上说，在机器学习方法中，它几乎是唯一能达到 superhuman，也就是超人类表现的方法。就是强化学习。

那强化学习技术到底怎么发生？如果从数学上展开会很复杂，但 Karpathy 总结得很直白。他对强化学习的表达比较“粗暴”，但并非不对。基本意思是：agent 做 action。让它在环境里做 action 后，只要某次 action 最终带来 reward，那么在 reward 到来之前做过的 action，都提升其概率。你可以把它看成最基本思想：做出动作后，一旦有 reward，就把直到 reward 到来前的所有动作概率都提高。

## 强化学习核心：Credit Assignment Problem    *12:43*

<span class="paragraph-timestamp" data-ts="12:43">12:43</span> **金成贤** 这时会出现一个有点奇怪的想法：这些动作里有些有帮助，有些没帮助，尤其随机行动时更是如此。那似乎只提高真正有帮助动作的概率才对。但这就是所谓 Credit Assignment Problem。它有点棘手，本质问题是：你怎么判断哪个动作真的有帮助？认真想会发现这并不容易。

就算是人，在没有先验知识时，出现了好结果，到底是哪个动作导致的？这个判断并不容易。人靠经验或推理去判断；如果只是把你丢进场景里，判断哪个动作有帮助非常难。人也在这方面经常犯错。

比如在没有规律的地方硬找规律，或者把完全无关的动作当成有效动作。这类情况很多，说明这确实是个普遍困难问题。赌博里人常犯这种错：以为有模式，以为某个动作有帮助，于是形成所谓“迷信”。

所以这个问题非常难。AlphaGo 让强化学习有过很多高光时刻，但在这个问题上，尤其 reward 很稀疏、要几百几千个动作后才给一次 reward 的场景里，它仍然很难。

强化学习本身是个难题。当年用强化学习解 Atari 游戏时，很多 Atari 游戏都被攻克，但其中有款叫 Montezuma's Revenge。就是截图里这个游戏。据我所知它到现在也只是平均人类水平，而且不用非常规手段（比如人类引导、环境重置能力）的话，应该还没达到 superhuman。所以对于不太适合强化学习的环境，也就是做了很多动作后才有 reward 的环境，仍然很难。总体来说并不容易。

## 为什么把强化学习引入 LLM：RLHF    *15:10*

<span class="paragraph-timestamp" data-ts="15:10">15:10</span> **金成贤** 所以关于 LLM，为什么引入强化学习、以什么形式引入，我先从这里讲。看起来最早是以 RLHF 形式主流化的。虽然在那之前也有一些不同形式，但最主流就是 RLHF。很多人争论 RLHF 到底算不算 RL，很多强化学习从业者说不算，但我觉得它算。基本想法是这样。

给 LLM 一个 prompt，让它生成两个回答。两个回答里会有相对好的和相对差的。然后由人来标注哪个更好、哪个较差。标注后根据这些结果训练一个 reward model。这个 reward model 接收回答并预测它是否“好”，尽量贴近人类评价。就是这样的模型。

而这个 reward model 在 RL 里扮演 reward function 的角色。做好 reward model 后，再让 LLM 生成多个回答，把这些回答输入 reward model，它会评估回答好坏。然后最大化这个评分，也就是做 RL 去最大化 reward。这样模型就会被训练成产出 reward 更高的回答，也就是更符合人类正向评价的回答。

这个过程就构成对齐。你训练模型生成人类偏好的回答，训练越多，对齐越强。RLHF 的基本思路就是这样。你可能会想：“为什么要这么做？”事实上因为很多人都这么想，所以也有很多时候不做。尤其开源模型里常出现“非得这样吗？”“为什么要这样？”的观点，所以经常不做。

## SFT 的局限与幻觉问题    *17:18*


<span class="paragraph-timestamp" data-ts="17:18">17:18</span> **金成贤** 这个问题原因很多，但最代表性的可能是 hallucination 问题。先说“为什么要做 RLHF”，你会想有没有别的方法。最典型的是 SFT。即由专家或人工写正确答案。

实际上甚至不一定是人写，常用已对齐模型比如 GPT-4 生成回答来构造正确答案。总之是专家给出正确答案。比如“Liechtenstein 的首都是 Vaduz”，我也是查了才知道。就这样构建数据。然后模型去模仿专家写的正确答案，仍然用 Next Token Prediction。输入是“The capital of Liechtenstein is”，训练它预测 token“Vaduz”。

这就是 SFT 的基本思路，很多人会想“真的需要 RLHF 吗？”“只做这个不行吗？”很多人是这样做的。实际上不少开源模型过去和现在都这样训练。

但关键是要看到：被训练的 LLM 和专家是不同 agent。LLM 是 LLM 这个 agent，专家是人，是另一个 agent。这两个不是同一个体。这个点必须认真处理。

来看会发生什么。假设模型被拿它本来就知道的问题训练。比如“The capital of France is Paris”，若模型本来就知道法国首都是 Paris。那么它在学什么？对于这个问题它已经知道答案，于是它学到的是“我就直接回答”的行为模式。

再假设是模型不知道的问题。Liechtenstein 的例子里 LLM 可能都知道，这里假设它不知道，用“The capital of Liechtenstein is Vaduz”来训练。模型不知道首都在哪，那它学到什么？当然会学到知识本身；但同时它也会学到“即使不知道也先回答再说”的行为模式。

这就是为什么 OpenAI 最近发 hallucination 论文时也提到类似点：模型会因为“给出回答”本身而被奖励。是同一件事。它学到“先回答”的模式。这个会显著改变模型：既要吸收新知识，还要学会不知道也回答。对模型的这种显著改动通常会带来坏结果，于是 hallucination 出现。在这种情况下，模型学到的是“就算不知道也先说点什么”。

## On-Policy 与 Off-Policy 学习概念    *20:15*

<span class="paragraph-timestamp" data-ts="20:15">20:15</span> **金成贤** 再深入想想为什么会出现这种问题。RL 里有 On-Policy 和 Off-Policy 的概念，很重要。Off-Policy 是学习 agent 和行动 agent 分离。行动 agent 通过行动产生数据，学习 agent 用这些数据学习。这就是 Off-Policy。通常 Off-Policy 难得多。最极端例子，我引用 Sutton 书里的：如果行动 agent 在做饭，学习 agent 不可能学会开车。它不能从这种数据里学到想学的东西。

所以两者是否对齐对学习难度影响很大。你可能会想“那为什么还要做？”“直接 On-Policy 不就行了？”但有些学习模式只有 Off-Policy 才能实现。专家有经验，专家生成数据，学生从中学习。比如老师亲身经历后积累经验，把过程总结成数据给学生，这不是更高效吗？

因为你可以学习另一个 agent 的学习成果和行动经验，所以数据效率更高。而 On-Policy 的约束非常强。过去的你和现在的你也可视作不同 agent。学习过程中 agent 持续变化，所以现在的自己很难直接用过去自己的经验。这就是数据效率上的巨大差异。

因此我们想做 Off-Policy，但 Off-Policy 很难。这个难题怎么体现？有个长期问题。典型 Off-Policy 形态是专家行动，模型学专家行动过程。用前面自动驾驶例子，人开完车、模型学人的驾驶过程，就是 Off-Policy 的典型例子。

但它学的是“人从 A 到 B 再到目的地”的过程。人有能力走这条路径，模型可能没有。于是模型在真实环境里可能去不了 B，跑到了 C。

到了 C 之后出现它从没见过的新状态，训练中没见过，于是就解不下去了。这个问题会和“模型本身是否具备该能力”绑定在一起。

给模型某个任务时，模型是否有能力解它，这是关键。如果模型没有能力，却按它“有能力”的前提去让它解题，现实里就会失败。

## 模型用于泛化的问题求解能力    *23:31*

<span class="paragraph-timestamp" data-ts="23:31">23:31</span> **金成贤** 在机器学习里，这可关联到 overfitting 概念，有相似性。我们常说 overfitting 是对给定数据拟合过度。比如这些点你都要连上，就会得到非常复杂曲线，但实际上更简单的直线可能泛化更好。教材级直觉通常是：数据变量少、模型简单，overfitting 可能性低。

但再深入一点，也可以这么看：发生 overfitting 时，每个数据点被完全记住了。可称为 memorization。这个“记住”很有意思。我们常说“模型把数据背下来了”。如果把 memorization 和 generalization 对照看，overfitting 状态可视为 memorization；而可泛化状态是超越记忆、抓住更本质模式。

有意思的是，这不只是“变量少、模型简单”这么简单。你给模型的问题是不是它能解，对结果影响很大。这里有个小图，假设做图像分类。把图做得很小，像素少，变量看似变少。那是不是像素越少越能防 overfitting？不一定。图太小的话信息没了。这里是一张苹果图，被我缩得很小。变量虽少，但已经无法解决真实图像分类问题。给“可解的问题”是关键。

从模型角度看也是：模型越大就一定更容易 overfitting 吗？也不总是。不能这么简单看。假设神经网络每一层只能做一次加法（可类比 attention 的某种受限操作），那每加一层就多能解一步。2 层可解两步加法，1 步和 2 步问题可以，3 步就解不了。

对模型解不了的问题，模型会怎样？如果网络太弱可能直接学不会；但神经网络通常很强，往往有能力记住全部数据。于是对这个问题很可能直接背题。怎么背？比如“出现数字 4 就输出 10”之类。就是原样记忆。对“解不了的问题”的记忆会导致一种 overfitting。这里其实需要 3 层。3 层意味着模型更大，但变大后 overfitting 反而下降。

所以所谓最佳 generalization，很多情况下是学到 algorithm。给它数字和答案时，它可以靠记忆大量模式学习；但最能泛化的是学会排序算法。也就是说，要在这个场景泛化，模型必须有能力学到排序算法。没有这个能力，就只能记模式，泛化就不可能。这意味着模型只有大到足以学到排序算法时，泛化才会出现。

再扩展一点：单层可执行操作有限。Transformer 一层（如 attention）通常只能做有限操作，层数又有限，所以单个 token 上可做的计算量有限。但若预测某个 token 所需计算量远超这个上限，那就是模型解不了的问题。我带了一个简单一元二次方程例子。假设模型不够大，无法一次性解这个方程。那模型就解不了，能做的就是记忆：“遇到这个方程就输出这个结果。”除非把全世界各种方程都记完，否则无法泛化。

但如果把所需计算拆到多个 token 上呢？也就是分步骤解，一步一步来。这样相比“一步到答案”，每一步计算量更小：先写公式、代入、逐步计算、最后化简。每个 token 每一步所需计算都小于一次性求解。于是变成模型可解问题。可解后，泛化就可能出现。如果模型对不可解问题只能记忆导致不能泛化，那么把问题拆成模型可解步骤后，它就能通过步骤学到 algorithm，学到 algorithm 就会产生 generalization。

<span class="paragraph-timestamp" data-ts="29:55">29:55</span> **卢正锡** 我开始理解你想走的逻辑了，挺有道理。请继续，这很有意思。

<span class="paragraph-timestamp" data-ts="30:02">30:02</span> **金成贤** 所以关于 generalization，让模型去解它能解的问题非常重要。这不只是用小模型、少变量；必要时你得把模型做大，必要时得增加 sequence length。

为了给模型一个它能解的问题。那时才可能 generalize。

## 预训练与搜索空间收缩    *33:38*

<span class="paragraph-timestamp" data-ts="33:38">33:38</span> **金成贤** 最终这些 LLM 获取这类能力是通过 pre-training，所以我先结合 pre-training 解释它们如何获得能力。先从这个讲。pre-training 在 RL 里作用非常关键。

假设我们要生成对应 100 个 token 的文本，假设这个问题大概 100 个 token 可解。那可能性数量就是 LLM 词表大小的 100 次方。以 Kimi K2 为例，词表约 163,840 个 token，也就是 163,840 的 100 次方。围棋搜索空间虽然也巨大，但这里远比它更大。

这是 Kimi K2 的训练 loss，我看图后大概是 1.32。1.32 换成 perplexity 大约是 3.7。这个 3.7 的含义是：每个 token 大约有 3.7 个候选。可以这么理解。

原本对所有 token 均匀赋概率时是 163,840 个选择。但经过 pre-training，选择数被大幅压缩，变成大概从 3.7 个里选 1 个。因为这是整段序列平均值，具体到上下文时选择数还会继续下降，在特别明显的 token 上会更少。很像 Library of Babel。

<span class="paragraph-timestamp" data-ts="35:17">35:17</span> **卢正锡** 所以“选择被减少”这个学习结果，意味着模型在无数随机路径中开始形成某些被定义和组织过的路径，可以这么理解吗？

<span class="paragraph-timestamp" data-ts="35:31">35:31</span> **金成贤** 对，可以用 Library of Babel 理解。所有可能的 100 token 序列，如果全组合，探索空间无限巨大。

<span class="paragraph-timestamp" data-ts="36:30">36:30</span> **金成贤** 再展开一点。我们常说 LLM 是通过预测 next token 训练的，这是常见说法。next token prediction 具体意思可以用很简单的词来想，先把它当 next word prediction。

## 通过强化学习涌现推理能力    *44:14*

<span class="paragraph-timestamp" data-ts="44:14">44:14</span> **金成贤** 那接下来问题是怎么把这种能力激发出来。多数情况下它被埋着，不容易浮现。浮现概率不是 0，但“不推理”的概率更高。最近有篇有意思的论文讲这个，我用它来解释直觉。有一个“生成推理”的概率，以及“推理时答对”的概率；也有“不生成推理”的概率。

而不推理时也有答对概率。对 LLM 来说，通常“不推理”概率更高。但一旦推理，答对概率更高。比不推理时更高。于是形成不对称状态。

那么在强化学习里会发生什么？虽然“生成推理”概率低，但相对其出现概率，它的正确率高。强化学习会提高正确答案概率，对吧？那生成了推理的样本就会被强化。

<span class="paragraph-timestamp" data-ts="45:22">45:22</span> **卢正锡** 所以方向上会激励它说得更长。

<span class="paragraph-timestamp" data-ts="45:27">45:27</span> **金成贤** 对，会朝这个方向。因为不对称，出现推理时会被更多关注，因为那时正确率更高。

即使推理概率很低，但相对这低概率它的正确样本很多，于是这些正确样本会被更强调。强化学习只看对错，所以“按正确答案评价”这件事本身意味着：即便发生概率低，只要正确样本多，就会得到超出其出现频率的更强 reward 强化。

<span class="paragraph-timestamp" data-ts="45:59">45:59</span> **卢正锡** 对，所以从前面说的 perplexity 角度，粗略说就是 token 上投入的 compute 越多，最终越好，大概可这么总结。

<span class="paragraph-timestamp" data-ts="46:12">46:12</span> **金成贤** 其实结果比这更好。按这篇论文的观点，相比在 pre-training 阶段学这个模式，强化学习能非常快地让它发生。

在每一步里出现概率都会上升。若某个 reasoning 序列“正确率高但出现率低”，它的出现概率会提升，而且论文描述是指数式提升，几何级增长。每一步几何增长，所以能力学得非常快。

## 强化学习成功的条件    *52:26*

<span class="paragraph-timestamp" data-ts="52:26">52:26</span> **金成贤** 我把前面讲的点总结一下。你可能感觉内容分了好几条线，所以这里重整一下。前面有个核心点：要让模型泛化，必须给模型它能解的问题。这是重要观点之一。也可表述为：要让它以可泛化方式解题，以模型能解的方式解题。也就是题目必须在模型能力范围内。

而且要 on-policy。为什么？因为 off-policy 时，专家解题所需能力模型未必有。那就像我前面画的，它被迫只看路径 B。实际上按模型自身能力应走路径 C，但它学不到，只学了 B。

可模型按自身能力走到 C 后就卡住了。这就是为什么要让模型在自身能力范围内解题，必须 on-policy。模型要自己解题、按自己的方式解，再拿 reward 学习。不是别人给它“你照这个解”的 guide 再去学。

<span class="paragraph-timestamp" data-ts="53:56">53:56</span> **崔升准** 所以必须通过自己的经验做 trial and error。

<span class="paragraph-timestamp" data-ts="53:59">53:59</span> **金成贤** 对，这样 generalization 才可能。因为它是在自己能力范围内做强化学习，不是用模型解不了的方式去解。而且不只是“能不能解”，模型也有自己偏好的解法，和人不同，所以必须 on-policy。

另外，给推理过程强加结构，比如像 MCTS 一样把问题分步并规定结构，这种结构其实是人的想法。是人类专家“这个问题应这样解”的假设。但它可能和模型真正能怎么解、会怎么解脱节。所以强加结构有时不必要，甚至有害。

我前面说过，pre-training 里推理概率很低，但这条推理路径正确率高，所以其概率会迅速上升。这个过程中关键是：你必须准确判对错。

对于那些不经推理、靠记忆解题且容易错的样本，错了就必须判错；而通过可泛化推理解出正确答案时，才给奖励。不能把错答案反馈成对。只有这样才能防止不可泛化模式被错误强化。

<span class="paragraph-timestamp" data-ts="55:39">55:39</span> **崔升准** 你上次讲 false positives 时说的就是这个。

<span class="paragraph-timestamp" data-ts="55:43">55:43</span> **金成贤** 通过准确判定正确答案，可泛化模式会相对得到更多强化。这些条件必须满足。

关于推理，如果存在一种“判分很严格、必须完全匹配才算对”的情形，并且先假设“把错答案判成对”这种情况不存在，

那相对高 reward 只会落在可泛化的样本上。可以这么看。

所以比较“可泛化 CoT 后给答案”和“不做 CoT 直接给答案”，后者在不可泛化场景会出现大量错误。因为这些错误，其占比会下降；而那些在多种问题上都能以可泛化方式给出正确答案的行为模式，会得到相对更强强化。

因为系统能准确把正确判为正确。当模型给出不可泛化答案并被误判为正确进而被强化时，这种不可泛化行为模式……

而“不要给这种行为强化”的角色就是准确反馈。它会抑制靠猜答对。如果判定准确，猜中并被强化的情况会相对被压制。

但如果判定不准，猜中并得到强化的概率会上升。那就会阻碍可泛化模式涌现。

所以给出准确反馈，有助于发现可泛化模式。可以这样理解。

## 结尾：On-policy RL 与人生：Jason Wei 的故事    *1:00:05*

<span class="paragraph-timestamp" data-ts="1:00:05">1:00:05</span> **金成贤** 关于 on-policy 和 off-policy。曾在 OpenAI、现在在 Meta 做研究的 Jason Wei 说过一句话。人类一开始也靠模仿学习。按老师教的方式学，或者拿自己认为好的示例去模仿，会通过模仿学到很多。

到了那个无法继续模仿的点，你只能 on-policy。你得自己尝试，积累经验，然后只能从中拿 reward。他是这么说的。

最终人总会面对类似困境。如果真有一个所有人都能照抄的目标，那复制它当然最好。

但因为每个人环境和能力不同，在这种环境与能力之间，要形成真正可泛化的模式，他说你只能通过自己行动去获取经验。这也是你必须超越模仿的原因。
