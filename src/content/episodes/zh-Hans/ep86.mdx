## 介绍 Backend.AI:GO    *00:55*

<span class="paragraph-timestamp" data-ts="00:55">00:55</span> **卢正锡** 那先请你简要介绍一下 Backend.AI:GO。然后在聊构建过程时，也请你从一线角度讲讲今天 agent coding 到底该怎么做。

<span class="paragraph-timestamp" data-ts="01:10">01:10</span> **Jeongkyu Shin** Backend.AI:GO 是什么呢，可能有些朋友已经知道了。我们过去 10 多年一直在做并提供一个 AI 基础设施操作系统，或者说一个操作框架，叫 Backend.AI。但问题是，它真正发挥效果通常要从 100 张 GPU 起步，100 到 1000 张 GPU 这个量级才会用到，所以对大多数人来说门槛很高。到 2024 年下半年，我们开始思考：灾害发生时，AI 怎么用？医院或金融机构通过云在用 AI，如果云挂了，这就是大问题。那就像应急发电机一样，像医院、金融楼地下室放的发电机那样，你在本地放 GPU 机器，外部 AI 挂掉时也能自主运行。我们就是为此开始做的。

于是我们做了一个 router。我们叫它 Continuum Router，并在 2025 年 3 月公开发布。NVIDIA 有个活动叫 GTC，今年我们也会去。大家可以来找我们。我们在 GTC 发布后反响很好。我们做了演讲，但继续做着做着，体量越来越大。我们的主要客户是企业，当时它有大概 19 个组件。安装规模非常大，整套部署下去，几乎可以做成很多人用过的 OpenRouter 这类服务，已经长到了那个级别。

所以我们意识到，这不是企业解决方案栈，而是服务栈。于是我们先暂停，把它暂时搁置，问自己：如果只能保留一件东西，留什么？最重要的还是 router。我们把 smart routing 抽出来，其他功能先放一边，决定把速度做到世界最快。于是从零重建，时间是去年 8 月，把 Continuum 项目里的 Continuum Router 部分单独抽出来重做，到 12 月做完。

我们决定要发布 1.0，并开始准备。它本质是个 router，这个 router 有完整的 converter 能力，也有各种灾害应对的 circuit breaking 功能。比如某方出问题了，我们先把它剔除；被剔除那一侧正在跑的模型会自然重路由到其他模型。类似功能很多，但真正要发布时发现没法好好展示。东西很强，但讲不清。于是我们觉得必须做 web UI，必须有可视化界面。

这和我们刚创业时很像。快 11 年了。当时我们说要做机器学习运行平台，别人问怎么做，我们就先开终端开始讲。敲这个命令，那时候 TensorFlow 还没出来，会看到 Caffe2 config，还会出现当时流行的 Theano。以前就是这么讲。但我们需要的是可视化。Chester，你当年也这么说过。于是我们做了一个教育平台 demo，后来还卖出去了。那也是在平安夜。

## Agentic Coding 的经验：Token 经济学与高速 Inference    *11:30*

<span class="paragraph-timestamp" data-ts="11:30">11:30</span> **卢正锡** 构建这 100 万行的过程，你之前做 agent coding 的方式和这次实战中的感受差异，以及因此变化的新流程、新方法论，你刚才也提到了。我们就从这里展开？

<span class="paragraph-timestamp" data-ts="11:49">11:49</span> **Jeongkyu Shin** 我觉得这也只是转瞬即逝的一个阶段。先讲一件事再往下。我们之所以开始做这个产品，是因为假期期间 Anthropic 做了双倍 token 活动，这成了导火索。起点就是“还能试什么”，然后就冲了。

但反过来看，可用 token 量直接和公司的竞争力绑定，尤其是 IT 公司。这是第一个教训。第二个教训是，当你把这么多 token 倾倒出去，人会给反馈，而在“人不直接开发”的产品里会出现几个瓶颈。

比如大约六个月前，merge queue 是瓶颈。开发速度太快，代码彼此冲突。但现在 merge queue 已经不是瓶颈了。merge queue 的解决也不是人做的，它们自己处理。我偶然做过一些测试，其中有一种情况是两个 AI 在同一份源码上并行做不同功能，最后功能也能正常完成。进展已经到了这个程度。

所以到了下一阶段，关键问题变成怎么少用 tokens。做同样任务时要提升性能，模型通常会选择 in-context learning 里常见的方法，也就是不可见的 thinking tokens，我们叫 thinking budget。演化方向一直是把这个量加大。增大当然会提升最终输出，但也会拖慢开发速度。所以问题是如何用更少的 thinking 去开发。thinking 越少，开发速度越快。

最终在大家都用 AI coding 的世界里，速度会变得极其重要。提速有两条路。第一条是生成更少 tokens 但结果不变，这是第一课题。第二条是让 token 生成本身非常快，这是第二条。也因此现在高速 inference 变得必要。不是 ChatGPT 那种我们熟悉的代码生成速度，而是能跑 5 到 10 倍迭代的超高速 inference 会非常关键。这是我经历后得到的两大结论。

<span class="paragraph-timestamp" data-ts="14:00">14:00</span> **崔升准** 就像几天前出来的 Codex Spark，对吧？

<span class="paragraph-timestamp" data-ts="14:02">14:02</span> **Jeongkyu Shin** 有意思的是 Spark 也上线了服务，我们在看的是同一件事。最终这场竞争会把需求推向高速 inference 市场。有钱的会用高速 inference 强化竞争力，钱不够的就会想办法让它少想一点。

在必须要性能的地方多想，不需要性能的地方，比如简单任务或 coding，让它少想。怎么应用这种自适应 thinking budget，或者怎么做能动态控制 thinking budget 的 harness。我觉得这个冬天大概会是这样。

<span class="paragraph-timestamp" data-ts="14:36">14:36</span> **卢正锡** 都是 trade-off，对吧？

## 生物 Tokens：AI 时代的认知负荷与多巴胺    *14:38*

<span class="paragraph-timestamp" data-ts="14:38">14:38</span> **卢正锡** 人会一直想这些。AI 都帮你做了，但等待时间会开始变得烦人。

<span class="paragraph-timestamp" data-ts="14:45">14:45</span> **Jeongkyu Shin** 等待时间确实是大问题，但对我个人来说，这个等待反而给了我思考机会。不是想 AI，而是开始想我自己。

<span class="paragraph-timestamp" data-ts="14:59">14:59</span> **卢正锡** 你最近把这叫做“bio tokens”。我们团队也有人说，多一点给 bio tokens 的时间挺好。

<span class="paragraph-timestamp" data-ts="15:07">15:07</span> **Jeongkyu Shin** 以我个人看法是这样。大概两年前，我开始用 ChatGPT 在一定程度上委托 coding，变成和它一起写。

然后从去年 4 月开始变化很剧烈，这样生活了大概 9 个月后，我白头发明显变多了，睡眠也变差了。高峰在 6、7 月，5 小时 refill 出来后，我不想浪费这 5 小时，就跑 3.5 小时，睡 1.5 小时，基本这么过。

慢慢地，这些也开始自动化。代码库本身大概 70 万行，总写入大约 120 万行。但 120 万这个数字对我个人非常有象征性。

我以前做 TextCube 项目时，3 年总共写了约 100 万行代码。但现在同样规模 40 天就写完了。

某种意义上，人生被压缩了。我花了约 1 个月做 Backend.AI:GO，做的过程中我会问自己，我真的只投入了 40 天吗？

回头看，感觉自己老了 3 岁。作为一个人。

## 软件史上的第三次大革命    *22:38*

<span class="paragraph-timestamp" data-ts="22:38">22:38</span> **Jeongkyu Shin** 现在确实是软件发生巨大迁移的时期。所以你会听到很多“必须做软件”的讨论。我们公司内部也经历了很多，开了很多研讨会，彼此讨论，思考之后该走哪条路。

我开场前也简短提过，这种变化以前也出现过几次。如果说我们没亲历的代际变化，从打孔卡、涂 OMR 到突然用键盘写代码，这是方法论级别的大变化。后面还有一些小变化，比如在编辑器里用方向键移动光标写代码，这在以前都不行。再到把多个源码文件组合起来构建程序。然后下一个大变化是智能手机到来。构建我们的 stack 时，以前核心是独立打包软件，也就是 package software。问题是怎么分发，介质是 CD、磁盘，还是过去说的 ESD（electronic software distribution），也有这些讨论，还有 freeware、先试后买的 shareware，以及 commercial software，就是商店里买的那种。

从这种分发概念转到智能手机和 web，它们间隔大约 10 年先后进入社会，通过网络分发，甚至不需要分发了。通过浏览器，就像远程安装的软件。也就是我们今天常说的 web service：远程运行的服务，以服务形态使用的软件。由此开发方式改变了很多。web server 变重要，支付系统、安全相关都出现了大迁移。

UX 方面也从键盘+屏幕中心，突然变成很小的设备，或者屏幕尺寸多样化设备，输入从物理键盘变成触屏键盘。现在这一轮，我觉得大概是第三次。

这里变化的是什么？我说过软件本身也在变。

但现在在变的软件里，我们通常说 software 是指创建 code，把 code 叫 software，而操作它的人大多是开发者。开发者来做，转向 web 后又出现了 ops 这个岗位。因为 web service 里开发和运维高度耦合，DevOps 诞生了，就是开发和运维一起做。

然后过程中 stack 也变化。以前大家都写代码，但进入服务化、web service 化后，出现了处理 backend/server 逻辑的人，也出现了负责用户实际接触界面、设计体验和行为的人，于是 frontend 成为独立岗位。再后来也出现了统筹一切做规划，或“我都能做”的 full-stack 等各种角色。

重点在这里。以前写代码占 70-80%，实现并运行服务栈占 20-30%。做手机 app 或 web service 大致分这两块。但至少我们的孩子如果今后不学这些概念，我觉得他们可能根本不会知道这些。

## 代码价值正在收敛到零吗？    *25:57*

<span class="paragraph-timestamp" data-ts="25:57">25:57</span> **Jeongkyu Shin** 因为现在我做 Backend.AI:GO 这个项目时，周五去参加了 roundtable，里面也聊到这个。roundtable 从上午 10 点到中午。10 点前我在咖啡店把内容过了一遍后交出去，等 roundtable 结束再看，已经提交了大概 22 个，也可能 21 个 PR，测试都做完并且全部 merge。如果这种速度成为常态，代码价值本质上会收敛到零。开发者在 DevOps 里做的事，像我之前说的，如果工作性质不彻底变化，就会失业或处境很艰难。但同时他们也会变得极其重要。

软件本身不会消失。看到有人说今天软件有危险，不是软件本身有危险。就像软件定义曾经从 OMR 打孔转到键盘 coding，现在是从键盘 coding 转向表达语义。coding 会继续分阶段演化。到那时，我认为产品核心价值会来自 engine。这里的 engine 指我们今天叫 code 的处理部分，本质上是 logic。code 本身不是目标。

但如果 code 的目的就是处理逻辑并让事情运转，那么未来这部分大多会由 deep learning 模型，或它的衍生模型来处理，未必一定是 Transformer。某种其他处理逻辑的 engine 会接管当前 code 的大部分职责，这是我现在的判断。

## 对“CS 无用论”的反驳    *1:22:19*

<span class="paragraph-timestamp" data-ts="1:22:19">1:22:19</span> **Jeongkyu Shin** 有时会出现“计算机科学无用论”，我觉得挺有意思。因为我 2000 年入学，本科是物理，同时修了计算机双学位。我上课时几乎没有真正科班计算机出身的教授。因为上一代里很多学校都还没有计算机系，所以做计算机教授的人，很多是其他系里大量用计算机的人把这门学科搭起来的。有化学、材料、物理背景的教授进入早期计算机系。现在教授大多是完整计算机系训练出来的人。

这意味着从诞生之初，计算机科学就有一半是方法论哲学。虽然我们区分 computer science 和 computer engineering，但本质上它是非常年轻的学科，相对于很多其他学科。也因此我觉得它变化会很快。它会一直是“教 Python 的系”“教 C 的系”“教架构和 OS 开发的系”吗？这些会继续教，因为非常重要，但不会永远保持现在这个形态。比如 neuro-computing、模型开发与理解模型核心、attention 架构是什么、如何构建不同系统，这些最近都大量进入课程。实际上“深度学习导论”这类课已经加入课程体系。从这个角度，最能快速适应变化的学科反而是计算机科学。我虽然是物理出身，但我仍这么认为。当然现在物理专业的人去处也更多了，这很好。

在这个过程中，有人说“AI 都替你做了，为什么还学计算机科学？直接让 AI 做不是更快吗？”但我真问行业里的人，他们更多担心的是反方向。比如因为 AI 什么都做了，“我的工作会不会消失？”电影行业也是，Seedance 都能生成那样的视频，那导演和摄影怎么办？这种危机感。沿着这个逻辑，我认为未来大概 5 年左右，IT 虽然已经深度融入并广泛使用，但会以极大规模进入整个社会核心。

短期看，大家说“为什么学编程，程序都让电脑写了”。但你在计算机科学学的不只是编程，而是构建逻辑的方法。最简单门电路逻辑如何搭建并演化成无数层逻辑，这种方法论或哲学，我觉得更接近在学这种思维结构。所以这种人也更容易快速学习其他领域。尤其在可以外包知识的时代，这点更明显。
