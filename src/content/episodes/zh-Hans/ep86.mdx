## Backend.AI:GO 产品介绍    *00:55*

<span class="paragraph-timestamp" data-ts="00:55">00:55</span> **卢正锡** 那先请您简单介绍一下 Backend.AI:GO，然后我们再进入它是怎么做出来的故事，并听听现在的 agent coding 到底该怎么做，来听听一线人士的 knowledge。


<span class="paragraph-timestamp" data-ts="01:10">01:10</span> **申正奎** Backend.AI:GO 是什么呢，有些人可能知道，我们做 Backend.AI 这个 AI infrastructure 操作系统、运营体系，已经做并提供了 10 多年。其实要真正用好它，通常要从大约 100 张 GPU 起，效果才明显。它是面向 100 张到上千张 GPU 的平台，所以普通人其实不太容易接触。

从 2024 年下半年开始，我们在想：如果发生灾害，AI 该怎么继续用。比如医院、金融机构把 AI 跑在云上，一旦云停了就会出大事。那这种情况下，能不能像备用发电机一样，在医院或金融机构地下机房放 GPU 机器，当外部 AI 挂掉时也能自己转起来。所以我们做了路由器。

我们把它叫 Continuum 路由器，并在 2025 年 3 月公开了它。NVIDIA 有个活动叫 GTC，我们今年也去，欢迎大家来。我们当时在 GTC 发布，反响很好，也做了演讲。但做着做着体量变得非常非常大。我们主要客户是 enterprise，这个组件大概有 19 个，安装起来太重，最后全套装完几乎已经到了可以搭一个像 OpenRouter 那样服务的级别。

所以我们觉得这更像 service stack，而不是 enterprise solution stack。于是先全部按下暂停，冷处理一下，想如果只能保留一个核心该保留什么。最后结论是：最重要的还是路由，那就把 smart routing 这部分拆出来，其他功能先放着，先把速度做到全球最快。于是从头重做，去年 8 月从 Continuum 整体项目里抽出 Continuum 路由器部分重做，然后 12 月做完，准备发布 1.0。

它本质上是路由器，但自带 converter 功能，也有面向灾害的各种 circuit breaking，比如某个节点出问题就临时判定为不可用，再把原本在那边跑的模型自然切到别的模型上。功能很多，但准备公开时发现没法讲清楚。东西很好，但解释成本太高。所以我们就想必须做 Web UI，得有可视化界面。

因为我们最早创业时也是这样，现在也快 11 年了。当时我们说要做机器学习运行平台，别人问怎么做，我们先打开终端讲。输入命令后，当时还没 TensorFlow，会先出来一堆 Caffe2 配置，也有当时流行的 Theano。后来发现光这样不行，得有看得见的东西。当时卢正锡也这么说过。所以我们还做过一个教育平台 Demo 来卖。那天正好是平安夜。

## Agent coding 的教训 - token 竞争力与高速 inference    *11:30*

<span class="paragraph-timestamp" data-ts="11:30">11:30</span> **卢正锡** 你做出这 100 万行的过程，以及你在此之前接触 agent coding 和这次实践中的感受，还有因此改变的流程和方法论，之前提到过。要不要从这里开始讲？

<span class="paragraph-timestamp" data-ts="11:49">11:49</span> **申正奎** 我觉得这也可能只是阶段性结论。先说一点：这个产品开始开发，是因为 holiday 季 Anthropic 做了 token 翻倍活动。于是就想着多做点什么，才开始。

反过来看，如果一个公司可使用的 token 量，特别是 IT 公司，会直接和竞争力挂钩，这是第一个教训。第二个教训是，当 token 大量喷出、用户反馈不断涌入，而开发本身几乎无人手工做时，会出现新的 bottleneck。

比如 6 个月前 bottleneck 是 merge queue。因为开发太快，彼此改动会冲突。但现在 merge queue 已经不是瓶颈了。merge queue 的处理也不用人做，它们自己处理。甚至我做过一个故意测试：两个 AI 在同一源码上争抢、开发不同功能，最后功能也都能正常落地。进步到这个程度了。

所以进入下一阶段后，核心变成“如何少用 token”。同样一件事，模型为了提高表现，通常会增加 in-context learning 里不可见的 thinking token，也就是 thinking budget。这个量变大当然结果更好，但同时也意味着开发速度变慢。

所以关键是怎么让它少 thinking 也能开发。少 thinking 本身就等于更快。而在“所有人都用 AI coding”的世界里，速度会很关键。提速方法有两条。

第一条是让它更少生成 token 但输出同样结果，这是第一个挑战。第二条是真正把 token 生成速度做得非常快，这是第二个方向。也就是现在需要 high-speed inference。不是我们熟悉的 ChatGPT code generation 那种速度，而是能做 5 到 10 倍 iteration 的超高速 inference 会变得非常重要。

这是这次做下来我最大的两个教训。

<span class="paragraph-timestamp" data-ts="14:00">14:00</span> **崔升准** 前几天出的 Codex Spark，感觉就是这个方向。

<span class="paragraph-timestamp" data-ts="14:02">14:02</span> **申正奎** 结果很巧，Spark 服务也开始了，大家看的是同一个方向。

最终这场竞争会出现高速 inference 的需求。钱够的人会用高速 inference 提升竞争力；钱不够的就得想办法让模型少思考。

只在需要性能的地方加大 thinking，不需要高性能、只是简单任务或编码的地方降低 thinking。怎么应用 adaptive thinking budget，或者怎么做能动态调这个 budget 的 harness。我觉得今年冬天大概会围绕这个展开。

<span class="paragraph-timestamp" data-ts="14:36">14:36</span> **卢正锡** 都是 trade-off。

## 生物 token - AI 时代人类认知负荷与多巴胺    *14:38*

<span class="paragraph-timestamp" data-ts="14:38">14:38</span> **卢正锡** 我也在想这个问题。因为都交给 AI 做了，等待时间开始让人烦躁。

<span class="paragraph-timestamp" data-ts="14:45">14:45</span> **申正奎** 的确，等待时间问题很大。但反过来，这段等待也让我有机会思考，不是思考 AI，而是思考我自己。

<span class="paragraph-timestamp" data-ts="14:59">14:59</span> **卢正锡** 最近大家叫这个“生物 token”。有人说，能用于生物 token 的时间变多了，挺好。

<span class="paragraph-timestamp" data-ts="15:07">15:07</span> **申正奎** 我个人看是这样。我前年开始把部分 coding 委托给 ChatGPT，和它协作。去年 4 月起变化巨大，这样过了约 9 个月。白头发变多了，非常多，而且越来越不睡觉。最夸张时，6、7 月那阵，5 小时额度一刷新就舍不得浪费，跑 3.5 小时、睡 1.5 小时，差不多这么过的。

后来这些也逐步自动化了。代码本身大概 70 万行，总写入大概 120 万行。这个 120 万对我有象征意义。以前做 TextCube 项目，我 3 年写的代码加起来差不多 100 万行。现在同量级代码 40 天写完。

某种意义上像把人生压缩了，一个月里把 Backend.AI:GO 写完。但写完回头看，我真的只付出了 40 天的努力吗？感觉像老了 3 年。

认知负荷不会下降。你再怎么把事交给 AI，认知负荷也不降，反而因为反馈不断涌来，人会变得非常疲惫。

但做的时候很爽。因为像游戏，会注入多巴胺，像现在流行手游抽卡，花钱抽角色，赢了就即时反馈，大家就喜欢。

用 agent coding 的过程，会基于速度给人某种快乐。因为以前对我是大工程，甚至感觉做不到的事，现在能达成。是我达成还是它达成有区别，但总之会给你那种多巴胺。

## 从软件历史看第三次大变革    *22:38*


<span class="paragraph-timestamp" data-ts="22:38">22:38</span> **申正奎** 现在肯定是软件发生一次大变化的时期。所以我们也不断听到“必须做软件”这类话，公司内部也做了很多 seminar 和讨论，思考接下来该走哪条路。开始前我提过，这种变化以前也出现过几次。

如果说是我们没亲历的代际变化，从打孔卡、OMR 标记突然变成键盘 coding，这是方法论层面的巨大变化。再往后的小变化，比如在编辑器里能用方向键移动光标写代码，现在可能很难想象，但当时做不到。包括把多个源码合并成一个程序这种方式。

再下一次大的变化是智能手机进入后，我们做 stack 时，原本核心是独立打包的软件包，重点是怎么 deliver，比如介质是 CD 还是磁盘，或者当时说的 ESD 电子分发。商业模式也有 freeware、shareware（先用后付费）、再到商用软件（去商场买）。

这种分发概念在智能手机和 Web 以约 10 年时差先后进入社会后改变了。我们通过网络 delivery，甚至在 Web 浏览器里连 delivery 都不需要，直接远程使用安装在远端的软件，也就是现在说的 Web 服务。随后开发相关的一切都大变：Web server 重要了，支付系统与安全体系重构过一轮。UX 层面也从键盘+大屏，变成小设备或屏幕尺寸变化设备，输入从物理键盘转到触摸键盘。现在这次按量级看，我觉得大概是第三次。

这次变的点是，软件本身一直在变。以前我们说软件就是代码，写代码、运营代码，大多是 developer 干。转到 Web 后，出现了要做 operation 的 ops；两者在 Web 服务里高度耦合，于是有了 DevOps。再后来 stack 分工也细化：原先人人全栈 coding，到了服务化 Web 形态后，出现了做后端逻辑和服务器的人，也出现了做用户界面、用户体验和行为设计的前端岗位。再往上还有统筹规划，或者“我都能做”的 full stack。分工越来越多。

核心是这个：以前写代码占 70\~80%，实现并运维服务 stack 占 20\~30%，我们按这两块开发移动 app 或 Web 服务。至少我觉得，未来我们的下一代如果不特意学，可能连这个概念都不知道。

## 代码的价值会收敛到0吗    *25:57*

## Cyber Formula 类比 - Claude Code vs Codex 的哲学差异    *1:07:49*

因为我是 00 级。我是物理系，也双修过计算机工程。那时上课，纯计算机工程出身教授并不多。因为更上一代本来就没这个学科。最早成为计算机工程教授的人，很多是其他学科里大量用计算机的人，把学问建起来。

这说明什么？计算机工程从一开始就有一半是方法论思想驱动。你也可以分 computer science 和 computer engineering，但本质上它是很新的学科，相比很多传统领域都新。所以我认为它变化速度也会非常快。

“学 Python 的系、学 C 的系、学架构和 OS 开发的系”这些会继续存在，因为非常重要。但我觉得不会一直以同一形态存在。比如 neuro computing，或者模型开发中模型核心是什么、attention 结构是什么、如何尝试其他机制，这些也已经大量进入课程。

实际上像“深度学习导论”也已经进了课表。所以从适应变化速度看，计算机工程可能是最快的学科之一。虽然我是物理出身，但我这么看。当然物理系最近出路更多，这也挺好。

在这个过程中会有人说，AI 都会做了，何必学计算机工程，直接让 AI 做不更快吗？但我身边同行更多的担忧反而是另一方向。比如 AI 什么都做了，并不意味着我没事干。就连拍电影，Seedance 都能把视频做成这样，那导演、摄影指导做什么？会产生这种危机感。

所以我认为未来约 5 年，IT 虽然现在已广泛使用，但会以更大规模深入社会核心。

短期看似“学编程有什么用，电脑都能写了”，但计算机工程里学的不是“程序”本身，而是搭建逻辑的方法。

从最简单门电路逻辑如何演进成复杂逻辑体系，这种方法论或说哲学、思维结构，才是核心。我认为更接近学这个。所以这类人再学其他领域也会更快。尤其在知识可外包的时代，会更明显。
