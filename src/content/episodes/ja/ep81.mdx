## MoE、より少ない計算でより高い性能    *04:34*

<span class="paragraph-timestamp" data-ts="04:34">04:34</span> **チェ・スンジュン** このグラフ、どう読めばいいですか？

<span class="paragraph-timestamp" data-ts="04:36">04:36</span> **キム・ソンヒョン** 実はこのグラフを入れるかどうか、かなり悩みました。直感的に理解するなら、一番薄い青の部分がいわゆるdense modelです。そして上の線たちがMoEモデル。そう見るといいです。ここは凡例が少し間違っていて、これらがMoEモデルです。

このグラフがとても重要でインパクトがあるのは、下の軸がtraining computeだからです。training computeが10の24乗くらいなら、フロンティアモデルより少し下のスケールです。この計算規模では、MoEモデルの性能はdense modelより7倍以上良い。つまり10^24 computeを使ったモデル同士で比較すると、MoEモデルはdense modelに対して、約7倍のcomputeを投入したdense modelと同等の性能を出します。

なのでdense modelに10^24 computeを使う場合と、同じcomputeでMoEを作る場合を比べると、MoEは約7×10^24相当のdense modelに近い性能を出します。これは非常にインパクトがあります。しかもtraining computeが増えるほどこの倍率が大きくなる。これは非常に珍しい現象です。モデルの世界では線形に2倍を維持するだけでも大発見ですが、現時点で分かっている範囲ではMoEはtraining computeが増えるほど倍率がさらに大きくなります。どんどん有利になる。dense modelと比べると、MoEを使わない理由がない段階です。使わないほうが不自然です。

<span class="paragraph-timestamp" data-ts="06:14">06:14</span> **チェ・スンジュン** でも、ここでMoEモデルが3種類に分かれているのはなぜですか？

<span class="paragraph-timestamp" data-ts="06:20">06:20</span> **キム・ソンヒョン** これはいわゆるsparsityです。総パラメータ数に対して、推論でトークン1個を予測するとき実際に使うパラメータの比率です。

<span class="paragraph-timestamp" data-ts="06:32">06:32</span> **チェ・スンジュン** つまり、より細かく分割される感じですか？

<span class="paragraph-timestamp" data-ts="06:35">06:35</span> **キム・ソンヒョン** はい。たとえばこれは総パラメータの4分の1だけ使います。

でも上のものは総パラメータの50分の1しか使いません。つまり実際に使うパラメータが減るほど、より疎になるほど、この倍率、いわばcompute multiplierが大きくなります。傾きが急になります。

もちろんこれはFLOPsで計算しているので、実際の推論条件やメモリ帯域などを考えると必ずしもこの通りにはなりません。ただ、純粋にtraining computeの観点でこのパターンが出るのはかなり驚きです。

<span class="paragraph-timestamp" data-ts="07:20">07:20</span> **チェ・スンジュン** 完全なscaling lawではないけど、似た雰囲気がありますね。

<span class="paragraph-timestamp" data-ts="07:24">07:24</span> **キム・ソンヒョン** 一種のscaling lawです。このscaling lawにも2つの要素があります。powerのexponentと、その前に付く係数です。exponentが変わるタイプのパターンだと、こういうことが起きます。この場合、これを使わないとおかしい、という技術進歩になります。MoEがその役割を果たしました。

2024年まではMoEモデルは稀でしたが、2025年に出てくるモデルはほぼMoEです。いわゆるedge device向けのケースを除けばほとんどMoEで、GPT-OSSのようなモデルもかなり疎なMoEでした。

そしてこのMoEのレシピを本当に確立したのはDeepSeekの大きな貢献です。ある意味でDeepSeekが設計したアーキテクチャが、前世代のLlamaアーキテクチャのようにベースアーキテクチャになりました。MoonshotのKimiなどは、DeepSeekアーキテクチャを改善しようとするのは不要だと言っていました。このアーキテクチャは十分に良いので、基本構造をそのまま使えばいい、と。そう言って実際にそのまま採用するような仕事をしていましたし、Mistralもおそらく似たことをしたと思います。それほどDeepSeekが切り開いたMoEアーキテクチャは確立されていて、これによってみんなが「このアーキテクチャを使えば、限られた計算資源でもGPT-4以上を狙える」と実感した。非常に重要な要素でした。

## RLVR、エージェント学習の新たな地平    *12:01*

<span class="paragraph-timestamp" data-ts="12:01">12:01</span> **キム・ソンヒョン** 次はRLVRです。これはo1の方法をほぼ完全に明らかにし、大きな変化を作りました。これも結局はDeepSeekがやったことです。

<span class="paragraph-timestamp" data-ts="12:14">12:14</span> **ノ・ジョンソク** まさにそうですね。

<span class="paragraph-timestamp" data-ts="12:15">12:15</span> **チェ・スンジュン** 考えると本当にすごいです。

<span class="paragraph-timestamp" data-ts="12:18">12:18</span> **ノ・ジョンソク** 彼らは本当にすごい仕事をしました。'24年末にDeepSeek-V3の論文が出てMoEなどを定義し、RLVRは2025年1月のDeepSeek-R1公開時に、その論文とともに出てきた方法論でした。

<span class="paragraph-timestamp" data-ts="12:35">12:35</span> **キム・ソンヒョン** 実はDeepSeekの影響力はむしろ過小評価されている気がします。これ自体がパラダイムと市場状況を大きく変えました。RLVR方式は、検証可能な正答を使ってrewardを与えることでreasoningを学習させます。2024年を通じて人々が探っていたものの核心に、想像できる中で最もシンプルな方法で最終的な決着を付けた。そしてこれがo1の手法に強いヒントを与えたことで、o1のようなreasoning modelの開発が可能になりました。

reasoning自体がモデル性能を大きく引き上げ、さらにいわゆるagent post-trainingの一側面も開きました。reasoningは単に数学問題を一生懸命考えることではなく、モデルがツールを使い、外部環境と相互作用し、その相互作用を通じてタスクをagentとして実行することです。

この全過程に対してRLVRの概念でrewardが与えられ、学習されると、モデル自体がagentとして学び始めます。これは非常に大きな影響があり、現在の市場にも影響しています。いま出てくるフロンティアモデルは、以前のようにbase pre-trainingをした後に少しだけpost-trainingする、instruction-followingを少し入れれば堂々と公開できる、という時代ではありません。

いまはagent post-trainingとreasoning能力が非常に基本的な機能として内蔵されています。これは本当に大きな変化で、モデルの実用性と可能性を大きく変えました。agent modelが実市場に与えた影響は、あえて強調する必要もないほどです。今出ているcoding agentやさまざまなagentは、すべてこのようなagent post-trainingで可能になった成果です。

<span class="paragraph-timestamp" data-ts="14:25">14:25</span> **チェ・スンジュン** ではこう言っても言い過ぎではないですか。RLHFがchatbotを作るためのpost-trainingなら、RLVRはagentを作るためのpost-trainingだと。

<span class="paragraph-timestamp" data-ts="14:35">14:35</span> **キム・ソンヒョン** はい、RLVRによってモデルをagentとして訓練できるようになった。そう捉えられると思います。

<span class="paragraph-timestamp" data-ts="14:43">14:43</span> **ノ・ジョンソク** これについては前回、ソンヒョンさんとかなり深く話しましたよね。本当に面白い回だったので、もっと気になる方はその前回回を聞くと良いと思います。

<span class="paragraph-timestamp" data-ts="14:56">14:56</span> **チェ・スンジュン** でも当時は「agent post-training」という言葉は使ってなかった気がします。


<span class="paragraph-timestamp" data-ts="15:01">15:01</span> **キム・ソンヒョン** そうですね、RLVRがagent性とどうつながるかは話していませんでした。たとえばこうです。モデルがagentとして機能するには、モデルにさまざまなツールを与える必要があります。たとえばeditorを使うためにtool callする、あるいはsource codeならcoding agentがrepository内のコードやファイルを読んで変更を書き込む。こうしたことはすべてツール経由で起きます。モデルはこれらのツールを通じて外界と相互作用します。この過程で、どうすればモデルにこれをうまくやらせられるか、どう訓練するか。こういうツールを使って目的タスクを実行できるようにどう訓練するかを考えると、RLVRは非常に簡潔な答えを与えました。まずモデルがツールを使えるようにしたうえで、どう使ったかの過程はひとまず脇に置き、最終結果で評価する。このパラダイムへのシフトです。

あるcoding taskについて、最終結果が満足できるかを評価します。そこを評価する。たとえばunit testは典型例です。unit testでコーディングが正しいか、最終結果を検証できます。すると最終結果に至るまでのプロセスは、いまは必ずしも人間が細かく設計しなくてもよくなります。

モデルが自律的にツールを使い、そのツール使用を通じてなんとか結果に到達する。望ましい結果に到達したらrewardを与える。これによってRLVRでagent post-trainingが起こります。もちろんこれはかなり単純化した説明で、cold start問題などさまざまな課題がありますが、基本アイデアはこれです。このように最終結果だけで評価することで、モデルをagentとして訓練可能になりました。以前ならこれらをすべて設計しなければなりませんでした。

<span class="paragraph-timestamp" data-ts="16:58">16:58</span> **チェ・スンジュン** このグラフは何ですか？

## RL深掘り：能力の創発か、組み合わせか    *18:22*

<span class="paragraph-timestamp" data-ts="18:22">18:22</span> **キム・ソンヒョン** これはモデル性能とは少し別の話かもしれませんが、RLに関して非常に興味深かったのはRLインフラの発展です。LLM RLは要求がとても煩雑です。モデルを訓練する必要があるのでtraining infrastructureが必要ですし、同時にそのモデルで生成して環境と相互作用する必要があるのでsampling engine infrastructureも必要です。sampling結果を通じて環境とやり取りし、環境から変化や結果を得るための基盤も必要です。

しかもこれらのインフラはすべて連結されています。training結果はsampling engineへ、sampling結果はenvironmentへ、environment結果は再びtrainingへ戻る。しかし実際のインフラ観点では、どれか一つだけでも非常に難しい。training部分だけ切り出しても、それを効率的にやること自体が難題です。pre-training infrastructureではほぼこれがすべてでした。

そこに加えて、非常に異質なインフラ同士が相互作用し始めました。モデル訓練は高速でなければならず、sampling生成も高速でなければならない。trainingとgenerationの切り替えも速く、評価とreward付与も速く、全体プロセスが正確でなければならない。この過程にわずかなエラーがあっても学習を妨げるという研究が多数出ています。

2025年を通じて、速さと正確さを同時に満たすのは常に難しいのですが、これが新しい問題を多く生みました。さらにMoEが主流化したことで、MoE RLをどう安定させるか、MoEモデルをRLでどう安定学習させるかも非常に重要なテーマになりました。非常に難しい問題ですが、中国側が正面から取り組んだからか、進展はとても速かった。これも2025年の重要な柱の一つだったと思います。そしてRL理解は大きく進みました。

RL理解の一軸は、おそらくRL手法自体の改善でしょう。RLは正確にどう起きるのか。RLが起きるとLLM内部で何が変わるのか。なぜLLMではRLがこれほどよく効くのか。こういう研究が大量に出ました。

その中で私が特に面白いと思った研究は、LLMやRLが出てきた初期からあった問いです。RLは新しい能力を与えるのか、それとも既存能力を引き出すだけなのか。つまり、もともと埋もれていた能力を掘り起こすだけなのか、という問題です。初期には「ほぼそれだけだろう」という見方が一般的でした。

しかしこの理解が深まる中で、RLが新しい能力を与えるという見方が出てきました。それがいわゆる「atomic skill」、原子的能力と、その原子的能力を組み合わせる能力です。atomic abilityは四則演算のようなものだと考えられます。こうした演算のようなものがatomic abilityで、それらをうまく組み合わせてより複雑な問題を解く能力はcomposition能力と見なせます。atomic abilityも重要で、組み合わせ能力も明確に重要です。今の理解では、atomic abilitiesはpre-trainingで学習される。演算のような能力はpre-training過程で学習される。一方でRLで学習されるのは、pre-trainingで学んだ能力を組み合わせる能力だ、と考えられています。

## self-playと内発的動機：人間と整合するAIの条件    *53:10*

<span class="paragraph-timestamp" data-ts="53:10">53:10</span> **キム・ソンヒョン** そしてRLに関連して必ず出るのが、いわゆる「self-play」です。AlphaGoがself-playで性能を高めた事例は人々に強い印象を残しました。

なのでそのself-playを通じて、これもデータ問題ですが、データ提供なしでモデルが自分で学ぶものを作れるのではという期待が高いです。

ただ、私たちが扱う数学問題やagent codingはGoのようなゲームではありません。zero-sum gameではないのでself-play実装は非常に難しいです。

たとえばこうです。問題を作るagentと、その問題を解くagentがいる。問題作成agentはより難しい問題を作り、問題解決agentはそれを解いて相互作用しながらモデルが改善される。こういう図は想像できます。

すると問題を書くagentは継続的にもっと難しい問題を作ります。難しい問題を作るほどrewardを得る。

でもここに罠があります。Goは分かりませんが、数学問題で考えると成功率0%の問題を作るのはとても簡単です。意味不明な問題を作ればいいだけだからです。

だから成功率0%の難問を作るのではなく、適度な難易度、成功率50%くらいの問題を作ろう、半分解ける問題を作ろう、となります。
